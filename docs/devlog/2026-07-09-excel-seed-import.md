## Что сделано

Разовый консольный сид `python -m scripts.seed_vendors` (`just seed`), грузящий 3 стартовых
Excel в живые таблицы ядра (`category`, `position`, `vendor`, `agreement`, `listing`),
идемпотентный, с `--dry-run`. Ничего вычислимого (светофор/звезда/проценты) не пересчитывает —
только раскладывает данные; аудит/`created_by`/инвариант ячейки ставят триггеры БД.

**Три слоя с жёстким разделением** ([backend/app/seed/](../../backend/app/seed/)):

- **`parse.py`** — чистые функции (str/int/None → dataclass, без openpyxl и БД):
  классификация строк (`classify_row`: heading/position/footnote/blank), разбор
  вендор-токена (`parse_vendor_token`: звезда `*`, приклеенный `Ujin`, скобки-область
  применения, кавычки не срезаем), классификация ячейки (`classify_cell`:
  прочерк/требование/список вендоров/смешанная), дерево категорий (`CategoryTree` —
  идентичность узла по числовому пути-кортежу, первое имя побеждает + предупреждение,
  проверка целостности).
- **`reader.py`** — openpyxl: `data_only=True`, единственный видимый лист, шапка по якорю
  `№` (не по фиксированной строке), границы колонок классов по шапке (НЕ по `max_column`),
  разворот объединённых ячеек для колонок классов на строках позиций. Классификация
  строк идёт по СЫРЫМ A/B (без разворота), чтобы горизонтальные merge заголовков не
  протекли в B.
- **`loader.py`** — `build_load(paths) -> LoadPlan` (DB-free сборка плана: reader + parse +
  дерево) и `execute`/`run` на async SQLAlchemy Core. Движок — `app.db.get_engine()`,
  транзакция `async with get_engine().begin()`.
- **`report.py`** — аккумулятор счётчиков + рендер отчёта + сверка с калибровкой
  (константа `CALIBRATION`, метод `RunReport.verify()`).

**CLI** ([scripts/seed_vendors.py](../../backend/scripts/seed_vendors.py)) — тонкий шим
(argparse → `asyncio.run(loader.run(...))`); флаги `--dry-run`/`--author`/`--freeze`/`--force`/`--verify`/`--yes`.
`just seed *args` (сид), `just seed-verify` (`--dry-run --verify`, ручная калибровка вне CI).

**Страховка `--yes` (добавлена по итогам инцидента).** Bare `just seed` без флагов теперь
**отказывает** (exit 2, к БД не подключается): боевая перезагрузка таблиц стандартов — только с
явным `--yes`; `--dry-run`/`--verify` разрешены без него. Причина: случайный `just seed` сразу
полез писать в основную Neon-БД; процесс замёрз в `idle in transaction` (QuickEdit-пауза
PowerShell посреди построчной вставки), откатили через `pg_terminate_backend` — durable-данных не
осталось, но guard закрывает сам класс ошибки.

## Находки разведочного скана (реализованы)

- **Маркеры требований** — только `Россия` (точное, `_REQ_EXACT`) и `По согласованию…`
  (префикс, `_REQ_PREFIX`). `ГОСТ` встречается лишь внутри названий продуктов — НЕ маркер.
- **Шапка не на фиксированной строке:** жилой/социальный — одноуровневая (стр. 5), офис —
  двухуровневая (группы стр. 6, классы стр. 7). Определяется по якорю `№` в колонке A.
- **Фантомные колонки:** офисный лист декларирует `max_column=16384` — границы колонок
  берутся по шапке (до первой пустой, с верхним пределом `_MAX_CLASS_COL`), а не по `max_column`.
- **Data-merge:** жилой `C89:H93` — один набор вендоров на 5 позиций × 6 классов; без
  разворота merge теряются вендоры 4 позиций.
- **Одиночная ячейка `Ujin`** — это вендор «Ujin», а не флаг интеграции (правило «вырезать
  Ujin» не применяется, если имя схлопнулось бы в пустое).

## Решения ревью

- **Дефолт `--no-freeze` (§13):** сид грузит только живое состояние; заморозку первого
  издания (`freeze_release`) админ делает из приложения после чистки данных. Механика
  freeze в коде оставлена и покрыта тестом.
- **Смешанная ячейка `Россия, <вендоры>` (§8.4):** разбирается как список вендоров, а
  ключевое слово требования пишется префиксом `Требование: Россия` в `note` каждого вендора
  ячейки (вендором не становится). Инвариант `listing_cell_chk` (ячейка — ЛИБО вендоры, ЛИБО
  одна мета-строка) не нарушается.
- **`--force` и защита проектов (§14):** сброс — `DELETE` в FK-порядке (дети → родители:
  `release_listing → release → listing → change_log → agreement_change_log → agreement →
  position → vendor_alias → vendor → category`), НЕ `TRUNCATE CASCADE`, и `compliance.*` не
  трогается. `--force` снимает только guard от перезаписи при существующих проектах; но FK
  `RESTRICT` из `compliance.project`/`project_selection` физически не даёт удалить строки
  ядра, на которые ссылаются проекты — `DELETE` падает, транзакция откатывается, проекты
  целы (свойство сильнее guard).

**Находка 2-го ревью — дедуп вендора В ПРЕДЕЛАХ ЯЧЕЙКИ** против уникального индекса
`uq_listing_cell_vendor`: повтор одного бренда в одной ячейке пропускается + предупреждение
с локацией, но звезда вендора агрегируется ДО skip (не теряется). На реальных данных
сработал: Ронсон/Newton Systems (офис), Flamco/СВОК (соц).

## Наблюдение по источнику (к сведению заказчика, НЕ дефект кода)

**Нумерация разделов НЕ консистентна между файлами** — один числовой путь означает разные
разделы:

- `3.2.2` = «Мебель жилая» (residential) vs «Мебель офисная» (office);
- `2.4.x` полностью расходится residential↔social (Бордюры/Облицовка/Озеленение vs
  Озеленение/МАФ/Ограждение);
- `1.1.8`/`1.1.9` сдвинуты (Механизированные парковочные системы / Система мусороудаления
  vs Система мусороудаления / Системы обслуживания фасадов).

Спека §6 (подтверждено ревьюером, §20.2) сознательно выбрала идентичность узла по **числовому
пути** + «первый файл (residential) задаёт имя» + предупреждение в отчёт; приведение к единому
виду — в источнике, не в скрипте. Следствие: в объединённом дереве позиции office/social для
общих числовых путей ложатся под метку residential. **Действие заказчика:** выверить нумерацию
в исходных Excel (или препроцессинг перед сидом). Сид честно печатает все конфликты имён в отчёт.

- Реальная запись на боевую БД и `--freeze` на 3 файлах — по решению админа из приложения
  после чистки данных (дефолт сида — только живое состояние).
- **Декомпозиция записи — ВЫПОЛНЕНО** (PR [#7](https://github.com/zhukovvlad/vendors_list/pull/7),
  [devlog](2026-07-09-seed-writepath-decomposition.md)). Построчные `INSERT … RETURNING`

  одной транзакции; добавлены `SET LOCAL statement_timeout`/`idle_in_transaction_session_timeout`.
  Атомарность §14 сохранена. Замер: реальный прогон ~13k строк — минуты → **≈31с**.
