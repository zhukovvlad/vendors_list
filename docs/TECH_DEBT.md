# Техдолг и отложенная работа

Живой реестр осознанно отложенных доработок: то, что **решено не делать сейчас**,
но нужно не потерять. Не баг-трекер и не бэклог фич — только уже принятые
компромиссы с указанием причины и где «болячка» проявится.

**Как вести:** добавляй пункт в нужную секцию с одной строкой «почему отложено».
Когда пункт закрыт — удаляй его (история останется в git и devlog). Источник
истины по деталям — спека/план соответствующей задачи (ссылки внизу секций).

## Дизайн-система (foundation)

Внесено при интеграции MR Design System (ветка `feat/design-system-integration`).
Foundation — только токены/темы/шрифты/реколор `button`; всё ниже сознательно
вне объёма. Источник: [спека](superpowers/specs/2026-07-08-design-system-integration-design.md)
(§«Вне объёма», §«Polish backlog») и [план](superpowers/plans/2026-07-09-design-system-integration.md).

### Шрифты
- **woff2-сжатие.** Самохостим сырые `.otf`/`.ttf` — тяжелее и без сжатия.
  Прогнать через woff2 (обычно −40–60% веса) при полишинге прода.
- **FOUT / `size-adjust`.** У Suisse и Inter (fallback) разные метрики → при
  загрузке шрифтов возможен сдвиг лейаута на первых кадрах. Лечится
  `size-adjust` в `@font-face` или `font-display: optional`.
- **Grtsk Giga без веса 400.** Самохостится в 100/300/500; дисплейный текст под
  `font-display` без явного веса подтянет ближайший (500 Medium). Для заголовков
  это ожидаемо — пункт-напоминание, чтобы никто не «чинил» фантом.
- **`font-synthesis: none` глобально.** Убирает синтетический жирный и в
  контентном тексте с пользовательской разметкой (`<strong>`/`<b>`). Для UI
  правильно; в контентных зонах с чужой разметкой включить синтез точечно.

### Компоненты
- **Focus-ring на `destructive`-кнопке.** Сейчас фиолетовый `--ring` на красной
  кнопке в фокусе — инородно. Дать destructive отдельный ring-цвет при полировке.
- **`--destructive-foreground`.** Определён в `:root`/`.dark`, но без маппинга в
  `@theme inline` и без потребителя (сплошная danger-кнопка использует литерал
  `text-white`). Добавить `--color-destructive-foreground` при появлении
  soft-danger компонентов — или убрать переменную до момента надобности.
- **Точные px-размеры кнопки.** Оставлены shadcn (`h-8`/`px-2.5`), не переведены
  в DS `11×24` — полиш под конкретные экраны.
- **`disabled` через `opacity`.** Оставлен shadcn-механизм (в тёмной поднят до
  `opacity-60`), не DS-реколор `border-strong`/`fg-subtle` — полиш.

### Чарты
- **Категориальная палитра.** `--chart-4/5` сейчас `provisional`; хендофф
  санкционирует для графиков только три хью (violet/mint/tan). Финал палитры на
  5+ рядов — дизайн-решение, финалим с реальными сериями на дашборде проектов.

## Сид-импорт Excel (разовый)

Внесено при реализации разового сида вендор-листов (ветка `feat/excel-seed-import`).
Источник: [спека](superpowers/specs/2026-07-09-excel-seed-import-design.md) (§8, §9, §11, §16)
и [план](superpowers/plans/2026-07-09-excel-seed-import.md); финальное ревью ветки.

- **Предупреждение о рассинхроне сетки групп офиса (§11) не реализовано.**
  `reader.ClassColumn.group_name` вычисляется, но `loader`/`report` его не потребляют —
  «громкое предупреждение» при расхождении сетки групп с `segment_group` (§11, помечено
  как не-блокер) отложено. Пока метки классов офиса совпадают с `segment.name` дословно,
  сетка сходится по позиции колонки; реализовать при первом реальном расхождении шапки.
- **Смешанная ячейка «По согласованию…, <вендоры>» молча роняет вендоров (§8.4).**
  `match_requirement` для `Россия` — точное совпадение (смешанная ячейка уходит в
  vendor-list + `Требование:` в note), а для `По согласованию` — префикс, поэтому
  «По согласованию…, Bosch» матчится как ячейка-требование целиком и `Bosch` теряется
  без предупреждения. В 3 стартовых файлах все `По согласованию`-ячейки — whole-cell,
  потерь нет. Закрыть, если источник начнёт смешивать `По согласованию` с вендорами
  (увести префиксные маркеры в ту же mixed-ветку, что и `Россия`).
- **Отчёт (§16) без части справочных счётчиков.** Не выводятся: число одиночных `Ujin`,
  список имён в кавычках, извлечённые note-области, эвристика «похожих» имён вендоров.
  Все — информационные; на корректность данных не влияют.
- **Метка freeze — ISO-дата.** `label` = «к Стандартам, 2026-03-25» вместо
  дотированной «25.03.2026» из спеки §13. Косметика; freeze по умолчанию выключен.
- **Эффект таймаутов сида не покрыт автотестом.** `_apply_timeouts` ставит
  `statement_timeout`/`idle_in_transaction_session_timeout`; тест проверяет их
  ПРИМЕНЕНИЕ через `SHOW` (ловит ошибку единиц измерения), но реальный обрыв по
  таймауту не воспроизводится без медленного оператора (`pg_sleep`) — сознательно
  не тестируем. Эффект верифицируется инспекцией.

## Матрица перечня

Внесено при реализации матрицы (ветка `feat/listing-matrix`). Источник:
[спека](superpowers/specs/2026-07-09-listing-matrix-design.md), [план](superpowers/plans/2026-07-09-listing-matrix.md).

- **Сворачиваемое дерево разделов.** v1 показывает плоские строки-заголовки по
  `category_path`; сворачиваемые узлы конфликтуют с серверной пагинацией — отложено.
- **Рекурсивные `category_path`/`category_sort_path`.** v1 сводит вызовы к
  O(различные категории на странице) через CTE `cats`. Материализация пути (доп.
  колонка на `category` с триггерным пересчётом / materialized view) понадобится,
  если категории окажутся почти уникальны per-position или дерево сильно вырастет.
- **Подсветка совпавших ячеек при поиске.** `q` по `vendor_name` отбирает позицию
  целиком, ячейки с совпадением не подсвечиваются (v1).
- **Групповая шапка над одной колонкой.** При сужении `segment_id` до класса,
  входящего в группу (офис), `_group_columns` вернёт группу из одного сегмента —
  групповая шапка над единственной колонкой. Косметика; схлопнуть при полировке.
- **`note` не показан на экране.** Эндпоинт отдаёт заметку по требованию
  (`MatrixCell.note`) и по вендору, но v1 их не выводит: вендорская `note` есть
  только как `title=`-тултип, требование-заметка вообще не показана. Вывести на
  экран, если понадобится.
- **`SET search_path` на функции пути (опциональное усиление).** `category_sort_path`
  (ревизия `0003`) — `LANGUAGE sql STABLE` без явного `search_path`, ссылается на
  `category` без схемы. Реального риска нет (единственный вызыватель — API через
  `read_conn`, без `SET search_path`), но best-practice — зафиксировать
  `SET search_path = public, pg_temp` НОВОЙ ревизией (`0003` уже накачена на живую и
  тест-ветку — не править существующий файл, правило CLAUDE.md §5).

## Дашборд «Обзор»

Внесено при реализации начального экрана (ветка `feat/dashboard-overview`). Источник:
[спека](superpowers/specs/2026-07-10-dashboard-design.md), [план](superpowers/plans/2026-07-10-dashboard-overview.md).

- **Триграммный fuzzy-детект похожих вендоров.** v1 (`count_merge_candidates`) ловит
  только коллизию нормализованного имени между бренд-ключами (консервативно: пропуск
  лучше шума). Опечатки/перестановки/`pg_trgm`-похожесть — отложено до потребности.
- **`merge_candidate_pairs` в отдельный ленивый эндпоинт.** Сейчас детект считается
  синхронно в `GET /dashboard` (устойчив к таймауту → `null`). Если детект станет
  тяжёлым — вынести в отдельный ленивый запрос, чтобы не задерживать метрики/черновики.
- **Токен `success`.** Не заведён — первый позитивный статус («всё чисто» сейчас через
  `text-muted-foreground`). Завести семантический `success` при первом реальном
  зелёном статусе (по образцу `warning` из этой ветки).
- **Отложенные действия дашборда инертны (развилка O4).** «Новый стандарт» и шевроны
  (черновиков и очереди внимания) рендерятся, но `disabled`/без навигации — их цели
  (редактор стандарта, карточки черновика/вендора) ещё не существуют. Активировать при
  появлении экранов-целей.
- **`just migrate-down` бьёт по основной БД, не по тест-ветке.** В отличие от
  `migrate-test`, рецепт `migrate-down` работает с дефолтным `DATABASE_URL` —
  проверка обратимости миграции на тест-ветке требует ручной осторожности (можно
  случайно понизить основную базу). Завести парный рецепт `migrate-down-test`
  (target=test), симметричный `migrate-test`.

Follow-up-минорки из финального whole-branch ревью (не блокеры, сознательно отложены):

- **Комментарий про инвариант в миграции `0004`.** `dashboard_open_drafts` коррелирует
  `listing` по `building_type_id` (через `segment`), а не по `release_id` — это верно
  только потому, что `uq_release_one_open` гарантирует ≤1 открытого релиза на тип. Без
  пояснения будущий читатель может принять это за пропущенный join на `release` и
  «починить». Добавить одну строку-комментарий в `_UP` рядом с LATERAL.
- **Слабые assertions в двух `last_touched_*` db-тестах.** `test_..._fallback_to_created_at`
  и `test_..._from_listing` проверяют только `is not None`, а не точное значение
  (fallback = `release.created_at`; из listing = `updated_at`/`updated_by`). Усилить до
  сравнения с ожидаемым значением — иначе перестановка `la.last_at`/`la.last_by` в
  будущем рефакторинге пройдёт незаметно.
- **Скелетоны только на карточках метрик.** Спека (§состояние загрузки) просит скелетоны
  на месте трёх карточек **и строк панелей** «Черновики»/«Требует внимания»; v1 рисует
  их только для метрик. Косметика (тест ловит лишь `.animate-pulse` > 0); добавить
  скелетоны-строки, если сдвиг лейаута при загрузке заметен.

## Карточка вендора

Внесено при реализации карточки вендора и кликабельности матрицы (ветка `feat/vendor-card`).
Источник: [спека](superpowers/specs/2026-07-11-vendor-card-design.md),
[план](superpowers/plans/2026-07-11-vendor-card.md), [devlog](devlog/2026-07-11-vendor-card.md).

- **Агрегация «Где разрешён» по brand-key на карточке владельца (O3).** v1 показывает только
  листинги самого вендора; вендоры, **которых представляет** этот вендор (через `represents`),
  не разворачиваются отдельным уровнем. Полная картина «где разрешены все бренды этой
  организации» требует дополнительного фронт-фильтра или API-эндпоинта. Отложено до
  потребности со стороны UI-требований.
- **Поток «Объединить» (O2) — из заглушки в диалог (merge-flow).** Кнопка в блоке
  «Бренд и объединение» показывает заглушку; полный процесс (выбор целевого вендора,
  перенос listing'ов, update `represents`) требует интерактивного диалога. Отложено до
  разработки экрана слияния вендоров.
- **Ленивое раскрытие «Где разрешён» по стандарту при сотнях позиций.** Блок `WhereAllowed`
  разворачивает дерево стандартов и листинги one-shot; вендор с сотнями позиций может
  лагировать начальный рендер. Отложено сворачиваемое дерево с ленивой загрузкой листингов
  per-standard до profile-метрик на реальных данных.
- **`PUT /vendors/{id}/agreement` без `response_model`.** Эндпоинт возвращает `dict[str, bool]`,
  но тело `{starred}` не типизировано в OpenAPI/клиенте. Ввести модель `AgreementToggleResult`
  в полировочном проходе.
- **`AliasCreate.alias` без `.strip()`**. Валидирует `min_length=1` но не тримит пробелы —
  прямой API-вызов с пробелом вставит пробельный alias (UI тримит). Косметика.
- **Модернизация DS `Card` + унификация поверхностей.** Репозиторный
  `components/ui/card.tsx` — старый вариант shadcn (`p-4`, `shadow-elevation-2`, без
  border-разделения секций, без `CardFooter`/`CardAction`/`--card-spacing`); карточка
  вендора вовсе использует локальную `CARD`-константу (`<section>`), матрица/дашборд —
  DS `Card`. Follow-up: обновить примитив `Card` до актуального shadcn и унифицировать
  поверхности трёх экранов (вид — rounded-lg+elevation vs плоский rounded-xl — решается
  один раз для всех). App-wide, вне полиша одного блока.

Follow-up-слайсы (следующие срезы):

- **C1 — чистка ~47 грязных имён после сида.** Сид парсит исходные Excel с опечатками,
  висящими кавычками, `(Native)` в именах, неконсистентными написаниями вендоров.
  Follow-up: разовая нормализация через ту же механику `PATCH /vendors/{id}` (rename→alias)
  с ручной сверкой человеком: грязное написание → гиперссылка на alias, чистое
  каноническое имя. Ловушка ТЗ §3.4 подразумевает маппинг вендоров через человека.
- **C2 — сегменты соцобъектов.** Тип «Социальные объекты» в сиде раскладывается в
  1 сегмент с именем самого типа (заглушка). Чип класса с именем типа при одном сегменте
  выглядит багом. Follow-up: продуктовая развилка — показывать ли одноклассовые стандарты,
  или скрыть класс при ровно одном сегменте. На карточке пока рисуется как есть.
- **C3 — a11y-полиш `InlineEditText`** (из финального ревью, все Minor). Доступное имя
  кнопки-дисплея = всегда `ariaLabel` («Редактировать имя»), а не видимый текст (имя вендора) —
  расхождение с WCAG 2.5.3 (Label in Name): голосовое управление и объявление скринридером не
  включают само значение. Плюс: после 409 инпут не возвращает фокус (был `disabled` во время
  сохранения, `autoFocus` уже отработал) — нужен `ref.focus()` в catch-ветке. Плюс: ошибка
  объявляется через `role="alert"` + `aria-invalid`, но без `aria-describedby`-связки инпута с
  текстом ошибки. Контракт «имя = `ariaLabel`» сейчас несущий (на нём держатся селекторы тестов
  Task 3/4), поэтому правка — отдельный срез, а не точечная замена.

### Режим правки карточки вендора

Внесено при реализации режима правки (view↔edit) и типографики v3 «Где разрешён»
(ветка `feat/vendor-card-edit-mode`, 2026-07-13). Источник:
[спека](superpowers/specs/2026-07-13-vendor-card-edit-mode-design.md),
[план](superpowers/plans/2026-07-13-vendor-card-edit-mode.md),
[devlog](devlog/2026-07-13-vendor-card-edit-mode.md).

- **Поток объединения вендоров.** Кнопка «Объединить» в edit-режиме — заглушка
  «в разработке». Мутации разрешений спроектированы общими (нейтральны к экрану),
  но перенос listing по `represents_id` + слияние — отдельный срез (см. также O2 выше).
- **Редактируемая сетка матрицы.** Listing-мутации (`add`/`exclude`/`restore`)
  vendor-scoped, но операция нейтральна — та же ячейка однажды правится из матрицы.
  Экран матрицы в этом срезе не трогали.
- **Редактирование `release.label` при публикации.** `ensure_open_release` (ревизия
  `0006`) ставит нейтральный «<тип> — рабочая версия»; `freeze_release` его не
  перезаписывает (0001:396-400). Уточнение label человеком — на будущем экране
  изданий (§5).
- **`add_listings` берёт `building_type` из `segment_ids[0]`.** Маркер open-релиза
  создаётся для типа первого сегмента; вызов, смешивающий сегменты разных типов, отметил
  бы черновиком только первый. Текущим UI недостижимо (позиция открывается в рамках одного
  типа). Усилить (per-distinct-bt `ensure_open_release` либо assert single-type), если
  появится мультитиповой вызов.
- **`_add_one_class` — гонка на пустой ячейке → 500 вместо 409.** live-check `SELECT`
  не берёт лок: два конкурентных add в одну пустую ячейку — второй ловит
  `uq_listing_cell_vendor` (SQLSTATE 23505), который транслируется не как 409 (это не
  `listing_cell_chk`/P0001), а как 500. Целостность держит индекс; на масштабе приложения
  практически недостижимо. Добавить `SELECT ... FOR UPDATE` или ретрай при 23505, если
  всплывёт.
- **Фидбэк ошибки на мутациях разрешений (exclude / restore / «+ класс»).** Вызовы
  `confirmExclude` (класс-`×`, диалог), `restoreListing.mutate` («вернуть») и
  `addListings.mutate` в поповере «+ класс» — fire-and-forget без `.catch` → при отказе
  мутации нет тоста ошибки (в отличие от переименования/«+ стандарт», где фидбэк уже есть).
  Для restore и «+ класс» 409 **достижим и покрыт бэкенд-тестом** (живая мета-строка в ячейке,
  `test_restore_meta_row_conflict_409`) — там тихий отказ вероятнее, чем на exclude. Happy-path
  покрыт; добавить `.catch → toast` всем трём для симметрии.
- **Комбобокс «+ стандарт» может устареть после исключения всего стандарта.** `useMetaPositions`
  не в `invalidatePermissions`, а `/meta/positions` отдаёт только позиции с живыми листингами.
  Исключение целого стандарта soft-удаляет последний листинг позиции, но комбобокс предложит её
  до переоткрытия. Самолечится (повторное добавление воссоздаёт листинг). Добавить ключ
  `["meta-positions"]` в инвалидацию либо смириться.
- **Сброс диалога «+ стандарт» на успешном закрытии.** Родитель закрывает диалог,
  выставляя `open=false` программно; у контролируемого Radix `Dialog` `onOpenChange` на
  программную смену пропа не срабатывает, поэтому сброс шагов (в `onOpenChange`) на
  success-пути не отрабатывает → при мгновенном повторном открытии видны прежние выборы.
  Инертно (добавленный стандарт становится `disabled`/present; худшее — повторный сабмит →
  409, уже с тостом). Сбрасывать по open-true или безусловно после успешного `onAdd`.
- **`aliasOpen`/`aliasDraft` не сбрасываются при выходе из edit.** Открытый инпут «+ вариант»
  переживает «Готово» и покажется снова при повторном входе в edit. Косметика; сбросить в
  обработчике тумблера режима.
