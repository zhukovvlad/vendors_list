# 2026-07-09 — Декомпозиция записи сид-импорта (batch + timeouts)

**Статус:** дизайн утверждён, готов к плану.
**Область:** `backend/app/seed/loader.py` (`execute`/`run`), db-тесты сида.
**Не в области:** интерактивный импорт §5 (staging→сверка→коммит), экспорт изданий,
нормализация имён/`vendor_alias`, COPY, committed-batches, staging-swap.

## Проблема

`loader.execute` (PR #5, merge-commit `5208e12`) пишет в БД **одной длинной
транзакцией** через ~14k построчных `await conn.execute(INSERT)`: 68 category +
743 position (RETURNING id) + 898 vendor (RETURNING id) + 117 agreement + ~13k
listing. Каждый INSERT — отдельный round-trip до Neon (us-east-1), исполняемый
последовательно с `await`, поэтому реальный прогон длится **минуты** и держит
широкое окно для `idle in transaction`.

Инцидент 2026-07-09: случайный `just seed` завис на 10 мин `idle in transaction`
(QuickEdit-пауза PowerShell посреди построчной вставки); откатили через
`pg_terminate_backend`. Guard `--yes` закрыл «случайный запуск», но не длительность
и не окно фризов.

## Цель

Минуты → секунды при **полном сохранении атомарности §14** (один прогон = полная
перезагрузка стандартов, всё-или-ничего) и всех инвариантов, закреплённых
db-тестами. Плюс операционная страховка от зависших транзакций.

## Решения (принятые в брейншторме)

- **A. Пакетные вставки** через asyncpg `executemany` (`conn.execute(text(INSERT),
  [list_of_dicts])`) — единообразно для всех 4 массовых таблиц. НЕ COPY (YAGNI на
  разовом скрипте; выигрыш доли секунды ценой доступа к raw-соединению) и НЕ
  multi-row VALUES (бьёт в лимит 65535 bind-параметров: 13k listing × 8 колонок ≈
  104k). `executemany` пайплайнит выполнения одного prepared statement — без
  round-trip на строку и без лимита параметров.
- **Предвыделение id** через `nextval` (см. §Поток, п.4) вместо `RETURNING`.
  Причина отказа от `RETURNING`-маппинга: порядок строк в `INSERT … RETURNING`
  формально **не документирован**; для `position` (нет натурального ключа) zip-по-порядку
  опирался бы на недокументированное поведение. `unnest … WITH ORDINALITY` не спасает —
  `RETURNING` в plain `INSERT` не видит `ordinality` из source-подзапроса.
- **B. Операционная обвязка** — `SET LOCAL statement_timeout` +
  `idle_in_transaction_session_timeout` в `run()` (не в `execute`), см. §3.

## Архитектура

`execute()` переписывается на **предвыделение id + 4 единообразных
batch-`executemany`**, всё в той же одной транзакции — атомарность §14 нетронута.
`run()` получает операционную обвязку. **Без изменений:** `build_load` (DB-free),
`_guard_no_projects`, `_reset`, `report.py`, CLI-шим, freeze-механика.

## Поток внутри `execute()` (новый порядок)

1. `set_config('app.user', :u, true)` — **как есть, остаётся ПЕРВЫМ**. Это
   бизнес-требование аудита: триггеры `listing_*`, `agreement`-change_log и др.
   срабатывают построчно и берут автора из установленного `app.user`. Переставлять
   нельзя.
2. `_guard_no_projects(conn, force)` — как есть.
3. `_reset(conn)` — как есть (каскад DELETE в FK-порядке дети→родители,
   `compliance.*` не трогаем).
4. **Предвыделение id** — только для 3 таблиц, чьи id нужны downstream: `category`
   (референс из `position.category_id` и `category.parent_id`), `position`
   (референс из `listing.position_id`), `vendor` (референс из `listing.vendor_id`
   и `agreement.vendor_id`). По одному запросу на таблицу:

   ```sql
   SELECT nextval(pg_get_serial_sequence(:t, 'id')) FROM generate_series(1, :n)
   ```

   В Python строятся карты: `path(tuple)→id`, `pos_key→id`, `name→id`. **Порядок
   возврата неважен** — нужны просто `n` уникальных значений, раскладываемых по
   объектам в любом порядке.

   - **`setval` НЕ нужен и добавлять его не следует.** Явные id взяты из того же
     sequence через `nextval`, поэтому sequence уже «впереди» вставленных значений
     — рассинхронизации нет. Классический рефлекс ревьюера «добавь `setval`» здесь
     был бы лишним кодом.
   - **`nextval` нетранзакционен:** запросы идут внутри той же транзакции, но их
     эффект на sequence при откате НЕ откатывается — значения «сгорают», дырки в id
     легальны. Это свойство, не баг.

5. **`category`** — один `executemany` с явным `id` + `parent_id` из карты.
   **Список параметров сохраняет порядок `tree.ordered()` (родители раньше детей).**
   `parent_id` FK проверяется построчно; порядок гарантирует валидность родителя —
   `DEFERRABLE` не нужен.
   > **Несущее допущение (закрепить комментарием рядом с кодом):** asyncpg
   > `executemany` выполняет наборы параметров строго последовательно в порядке
   > списка (последовательный пайплайн одного prepared statement) — именно это
   > заменяет deferrable для `parent_id`. Если кто-то позже «оптимизирует»
   > категории на multi-row VALUES с чанками или на параллельные вставки, FK-валидность
   > родителей тихо сломается.
6. **`position`** — один `executemany` с явным `id` + `category_id` из карты.
7. **`vendor`** — один `executemany` с явным `id`. **`agreement`** — отдельный
   `executemany` по звёздным вендорам (`vendor_id` известен из карты; собственный id
   agreement downstream не нужен — остаётся на default sequence). `agreement`-change_log
   триггер работает построчно и при `executemany`, берёт автора из `app.user`
   (см. п.1).
8. Карта сегментов `(code, segment_name)→id` — как есть (SELECT засеянного
   справочника `segment`⋈`building_type`).
9. **`listing`** (~13k) — один `executemany`; все FK (`position_id`, `vendor_id`,
   `segment_id`) известны заранее из карт. Без `RETURNING`. Триггеры
   `listing_stamp`/`listing_audit`/`listing_cell_chk` срабатывают построчно и при
   `executemany` — аудит и инвариант ячейки сохранены.
10. **`freeze`** (опц., дефолт выкл) — как есть: ≈3 `release` (по одному на
    представленный building_type), row-by-row с `RETURNING` + `freeze_release(:rid,:a)`.
    Объём ничтожен — батч ради трёх строк не городим.

## Операционная обвязка (B) — только в `run()`

Хелпер `_apply_timeouts(conn)` (две строки `SET LOCAL`), вызывается в `run()`
внутри `async with get_engine().begin() as conn:` **до** `execute`:

```sql
SET LOCAL statement_timeout = '60s'
SET LOCAL idle_in_transaction_session_timeout = '15s'
```

- Литеральные константы (не пользовательский ввод) — инъекции нет. `SET LOCAL`
  живёт только в текущей транзакции.
- **Комментарий-инвариант рядом с `_apply_timeouts` (обязателен):** idle=15s
  безопасен, потому что `build_load` (весь парсинг Excel) в `run()` исполняется
  **ДО** `begin()`; внутри транзакции клиентских пауз нет, только пайплайн вставок
  с мс-зазорами. Перенос парсинга внутрь транзакции сделает 15s миной.
- `statement_timeout=60s` де-факто ограничивает `_reset` (каскад DELETE) и
  `freeze_release`. Для `executemany` в extended-протоколе PG таймер гасится
  завершением каждого `Execute`, т.е. 60s применяется к **каждому** выполнению
  statement'а, а не к пайплайну целиком — запас огромен.
- Только на боевом пути: `--dry-run`/`--verify` выходят из `run()` ДО открытия
  транзакции, timeouts их не касаются (подтверждено по коду `run()`).

## Обработка ошибок / атомарность §14

Семантика без изменений: любой сбой (FK `RESTRICT` из-под проектов, нарушение
`listing_cell_chk`, таймаут, обрыв) → исключение → `begin()` делает ROLLBACK →
«всё-или-ничего». Наполовину залитые стандарты невозможны. Предвыделенные `nextval`
при откате сгорают (дырки в id легальны). Guard проектов и `--force`-семантика
(снимает guard, но FK `RESTRICT` из `compliance` физически не даёт снести
привязанное ядро) — нетронуты.

## Тестирование (инвертированный TDD, маркер `db`, тест-ветка Neon)

**Регрессия — существующие 6 db-тестов проходят БЕЗ правок.** Они адресуют строки
по именам (`WHERE name = 'ТеплоСила'`, сегменты через `get_segment_id`) и по
количествам (идемпотентность), конкретных значений id не закрепляют — «плавающие»
предвыделенные id им безразличны. Это регрессионная сеть на идентичность результата
батч-пути: автор через триггер, `vendor_starred`, guard проектов, FK-RESTRICT на
`--force`, идемпотентность, freeze-снимок.

**Новые db-тесты** (закрывают ровно то, что вводит декомпозиция):

1. **Sequence-sync после `execute`.** Прогнать `execute`, затем вставить новую
   строку `vendor` (и/или `position`/`category`) БЕЗ явного id через фабрику →
   PK-коллизии нет. Доказывает, что sequence «впереди» предвыделенных значений —
   закрывает рефлекс «нужен `setval`» доказательством, а не словами.
2. **Вложенные категории.** План с родитель→ребёнок (2 уровня) → `parent_id`
   разрешается верно, порядок `executemany` сохраняет FK-валидность. Закрывает
   несущее допущение из п.5 (последовательность executemany вместо deferrable).
3. **Применение timeouts.** Db-тест внутри обычной fixture-транзакции: вызвать
   `_apply_timeouts(conn)` → `SHOW statement_timeout` / `SHOW
   idle_in_transaction_session_timeout` → сравнить значения. Ловит класс ошибок
   единиц измерения: `SET statement_timeout = 60` — это 60 **миллисекунд**, не
   секунд; опечатка при рефакторинге превратила бы боевой сид в машину по обрыву
   транзакций, и инспекция кода это надёжно не удержит. Цена — ~10 строк.

**Эффект** таймаутов (реальный обрыв по медленному оператору) автотестом НЕ
покрываем — нужен `pg_sleep`-оператор, овчинка не стоит выделки. Формулировка для
дока/девлога: «эффект таймаутов верифицируется инспекцией; применение и значения
закреплены тестом через `SHOW`».
