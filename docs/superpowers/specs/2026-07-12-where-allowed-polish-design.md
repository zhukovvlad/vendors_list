# Полиш блока «Где разрешён» на карточке вендора — Design

**Дата:** 2026-07-12
**Ветка:** `feat/where-allowed-polish` (от `main`)
**Статус:** дизайн согласован с заказчиком → готов к плану
**Предшественник:** `feat/vendor-card-polish` (PR #21: редизайн карточки + инлайн-правка шапки).

## Контекст и проблема

По живому экрану заказчик отметил: блок «Где разрешён» на карточке вендора плоский и
невнятный. Это целевой полиш **только этого блока**; остальную карточку (шапка, алиасы,
бренд/объединение) не трогаем.

Диагноз (по живому скрину и референс-макету заказчика, тёмная тема):

1. **Нет иерархии уровней.** Строка стандарта и имя позиции визуально почти одинаковы,
   отступ позиций едва заметен, вертикальной направляющей не видно — глаз не сегментирует
   границы стандартов.
2. **Шум повторяющихся чипов.** Когда вендор разрешён во ВСЕХ классах позиции, перечень из
   4–6 идентичных чипов не несёт информации и повторяется на каждой позиции.
3. **Легенда выглядит как поле ввода** (рамка + подложка на всю ширину).
4. **Фокус-обводка** на строке аккордеона слишком громкая.

Референс-макет: `where_allowed_block_v2.html` (сырые hex тёмной темы — сопоставлены на
семантические токены DS ниже).

## Ключевая находка (нулевой шаг: сверка контракта)

Правилу «все классы» нужен **знаменатель** — сколько всего классов (сегментов) у типа
объекта. Текущий payload его **не содержит**: функция `vendor_where_allowed(vendor_id)`
([0005](../../../backend/migrations/versions/0005_vendor_where_allowed.py)) отдаёт строки только
для `keys` — сегментов, где вендор либо жив-`allowed`, либо был в снимке последнего
published-релиза (`excluded`). Сегменты типа, где вендора никогда не было, в выборку не
попадают. То есть payload даёт числитель (в скольких классах вендор разрешён), но не
знаменатель. Значит нужна **минимальная добавка на бэкенде**.

Определение знаменателя — **число сегментов `building_type`** (позиционных ограничений
сегментов в схеме нет: `listing` — единственная связь вендор×позиция×сегмент, отдельной
таблицы применимости «позиция→сегмент» нет). Универсум классов позиции = все сегменты её
типа объекта.

**Живость сегментов (разобрано по DDL):** таблица `segment`
([0001](../../../backend/migrations/sql/0001_core_schema.sql), строки 56–66) не имеет
`deleted_at`/`is_active`/статуса — только `id, building_type_id, group_id, name, sort_order`.
Поэтому агрегат `count(*)` по `segment` не требует фильтра живости. (Если такой флаг появится
в будущей ревизии — агрегат придётся сузить; на момент дизайна его нет.)

## Решения

### A. Контракт (бэкенд) — минимальное расширение

- В схему `WhereAllowedStandard`
  ([schemas/__init__.py](../../../backend/app/schemas/__init__.py)) добавляется поле
  **`segment_count: int`** — всего классов (сегментов) у `building_type` данного стандарта.
- В эндпоинте `get_where_allowed`
  ([routers/vendors.py](../../../backend/app/routers/vendors.py)) после сборки `standards` —
  один агрегат по типам, которые реально встретились в выборке:

  ```sql
  SELECT building_type_id, count(*) AS n
  FROM segment
  WHERE building_type_id = ANY(:ids)
  GROUP BY building_type_id
  ```

  результат раскладывается по `standards[*].segment_count`. Типы, которых нет в выборке
  вендора, не запрашиваются (пустой блок «Где разрешён» → пустой список, второго запроса нет).
- SQL-функцию `vendor_where_allowed` **не трогаем**, миграций БД **нет**. После правки
  контракта — `just types` (регенерит `schema.d.ts`, gitignored).
- Golden-rule #6: «все классы» — презентационный роллап, не инвариант БД; поэтому в БД он не
  переносится. В API отдаём только сырой знаменатель, само правило — на фронте.

### B. Правило «все классы» (фронт, презентационно)

Чистый юнит-тестируемый хелпер в
[model.ts](../../../frontend/src/screens/vendors/model.ts) (без рендера). Для позиции:

- **«все классы»** ⟺ `allowed_count == segment_count` **И** `excluded_count == 0`
  → вместо перечня один **тихий контурный чип «все классы»** справа от имени позиции, в одну
  строку (контур `border-border`, текст `muted-foreground`, без подложки — визуально тише
  обычных allowed-чипов, у которых `bg-accent`).
- **иначе** (частичный набор и/или есть excluded) → текущий перечень: allowed-чипы (`bg-accent`)
  + зачёркнутые excluded-чипы (пунктир `border-border-strong`, `line-through`, тултип
  `excludedTooltip` — как реализовано в PR #21).

Тонкий кейс: «полное покрытие, но есть excluded» → всё равно перечень (исключение **никогда**
не прячется за сводкой «все классы», т.к. условие требует `excluded_count == 0`).

Роллап стандарта: если **все** его позиции = «все классы» → в заголовке-полосе выводится
сводка **«N позиций · все классы»**, и такой стандарт по умолчанию **свёрнут** (детали
не нужны — заголовок самодостаточен).

Побочный эффект: снимается визуальный симптом **C2** (одноклассовый тип «Социальные объекты»
с покрытием → «все классы» вместо чипа-заглушки с именем типа). Корень C2 (данные-заглушка
сида) остаётся в TECH_DEBT.

### C. Визуальная иерархия

- **Заголовок стандарта — утопленная полоса** во всю ширину блока: `bg-muted` (sunken),
  волосяные `border-t border-b border-border`, имя 13px medium, счётчик справа
  (`muted-foreground`).
- **Leading-шеврон** слева от имени: открыт → `text-primary` + поворот; закрыт →
  `muted-foreground`. Реализуется **кастомным триггером на `AccordionPrimitive.Trigger`
  (Radix напрямую)** с собственным шевроном, состояние — через `data-state` (open/closed).
  Штатный `components/ui/accordion.tsx` **не трогаем** (ui-слой shadcn остаётся стоковым,
  не расходимся с генератором при обновлениях; специфика живёт в экране).
- **Позиции** — с отступом от полосы (`ml-3`) и **левой вертикальной направляющей**
  (`border-l border-border` + `pl-4`).
- **Разделители между позициями** — тончайшая линия `border-t border-border/60` (альфа-модификатор
  существующего токена; тише границ карточек; ничего нового не заводим).

### D. Легенда + фокус

- **Легенда** — тихая строка текста под деревом, **без рамки-коробки и подложки** (не должна
  читаться как input): мини-образец зачёркнутого чипа + «— был в последнем релизе, исключён ·
  показано текущее состояние стандартов». Показывается **только при наличии excluded** —
  переиспользуем существующие `whereAllowedLegend`/`hasExcludedChips` из
  [model.ts](../../../frontend/src/screens/vendors/model.ts) (без дублирования логики).
- **Фокус** — штатное тонкое `focus-visible`-кольцо (`--ring`) по границе полосы-заголовка,
  без толстой обводки на всю строку. Единообразно с фокусом аккордеонов в остальном
  приложении.

### E. Поверхности (отложено, follow-up в TECH_DEBT)

В этом срезе поверхности блока остаются как есть — `<section className={CARD}>`
(`rounded-xl border border-border bg-card`, плоско). **Не** переводим на DS `<Card>` в этом
срезе, потому что:

- Карточка вендора использует бес­по­доб­ную (bespoke) поверхность, а репозиторный
  [card.tsx](../../../frontend/src/components/ui/card.tsx) — **старый** вариант shadcn
  (`Card/CardHeader/CardTitle/CardContent`, `p-4`, `shadow-elevation-2`, без border-разделения
  секций, без `CardFooter`/`CardAction`/`--card-spacing`). Матрица и дашборд используют его же.
- Актуальный shadcn Card разделяет секции бордерами (`border-b` на `CardHeader`, `border-t` на
  `CardFooter`, `--card-spacing`). «Сделать как в генераторе» = **модернизировать DS-примитив
  Card** — а он общий для трёх экранов; это app-wide работа со своим объёмом.
- Внутренняя иерархия блока (полоса-заголовок + направляющая + разделители позиций) — это
  вложенное дерево «стандарт→позиция», а не header/footer-хром карточки; она строится
  собственными бордерами (решение C) **независимо** от того, `<section>` снаружи или `<Card>`.

**Follow-up (TECH_DEBT):** модернизация DS `Card` до актуального shadcn (border-разделение
header/footer, `--card-spacing`, `CardFooter`/`CardAction`) + унификация поверхностей матрицы,
дашборда и карточки вендора на единый вид — отдельным срезом, решение о виде поверхности
(rounded-lg + elevation vs плоский rounded-xl) принимается один раз для трёх экранов сразу.

### F. Тесты

- **Юниты правила «все классы»** (`model.ts`, без рендера):
  (а) полное покрытие без excluded → «все классы»;
  (б) полное покрытие, но есть excluded → перечень (исключение не спрятано);
  (в) частичное покрытие → перечень;
  (г) одноклассовый стандарт (`segment_count == 1`) с покрытием → «все классы»;
  (д) стандарт, где ВСЕ позиции «все классы» → сводка «N позиций · все классы» в заголовке.
- **Смоук вёрстки** (`VendorCardScreen.test.tsx`): полоса-заголовок (sunken `bg-muted`),
  левая направляющая у позиций, легенда без рамки-коробки, leading-шеврон и его состояние по
  `data-state` (open/closed).
- **Контракт** (backend): api-тест на присутствие и корректность `segment_count` в ответе
  `GET /vendors/{id}/where-allowed` (полный тип vs частичное покрытие); затем `just types`.

### G. Границы среза

Только блок «Где разрешён» на карточке вендора. **Не** трогаем: шапку, алиасы,
бренд/объединение, список вендоров, поток объединения (O2), C1 (чистка имён), корень C2
(данные соц-сегментов), модернизацию DS `Card` (→ E, follow-up).

## Токены (сопоставление макета)

`bg-card` (#16121F), `bg-muted` (#110D1C, sunken-полоса заголовка), `bg-accent` (#1E1930,
подложка allowed-чипа), `border-border` (#2A2640, волосяные линии/направляющая),
`border-border/60` (разделители позиций), `border-border-strong border-dashed` (#3A3556,
пунктир excluded), `text-primary` (акцент шеврона open), `text-muted-foreground` (вторичный
текст/счётчики/шеврон closed), `text-foreground` (имена), `--ring` (фокус). Ничего нового не
заводим.

## Архитектура и изоляция

- **Бэкенд:** одно поле в Pydantic-схеме + один агрегат в существующем эндпоинте. Никакой новой
  логики правил (правило — на фронте); функция `vendor_where_allowed` неизменна.
- **Фронт:** правило «все классы» — чистая функция в `model.ts` (тестируется без DOM). Полоса-
  заголовок стандарта — небольшой презентационный подкомпонент на Radix-триггере, живёт в
  `VendorCardScreen.tsx` (или рядом), не меняет DS. Легенда/чипы переиспользуют существующие
  хелперы. Каждая единица: понятна изолированно, testable, с чётким интерфейсом.

## Порядок реализации (ориентир для плана)

0. (Разобрано в дизайне — DDL сегмента без soft-delete.)

2. Правило «все классы»: хелпер в `model.ts` + юниты (5 кейсов).
3. Вёрстка иерархии: полоса-заголовок (Radix-триггер, leading-шеврон), направляющая,
   разделители, чип «все классы», сводка в заголовке + смоук-тесты.
4. Легенда (без рамки) + фокус (тонкое кольцо).
5. Финал: `just ci` зелёный, devlog, CLAUDE.md (карта/§5 при необходимости), TECH_DEBT
   (follow-up модернизации DS `Card`; C2 остаётся), PR в `main`.
