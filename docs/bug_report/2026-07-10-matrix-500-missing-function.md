# Матрица: 500 на старте — `function category_sort_path(integer) does not exist`

**Дата:** 2026-07-10 · **Ветка:** `feat/listing-matrix` · **Серьёзность:** блокер dev-стенда
**Файлы:** `backend/migrations/versions/0003_category_sort_path.py`, живая база Neon

## Симптомы

Экран матрицы не открывался вообще: `GET /listings/matrix` → 500. В
`backend/logs/errors.log` — точный диагноз прямым текстом:

```
asyncpg.exceptions.UndefinedFunctionError: function category_sort_path(integer) does not exist
```

## Расследование

Короткое — лог сразу назвал отсутствующую функцию. Вопрос был не «что», а
«почему её нет»: миграция `0003_category_sort_path` существовала, тесты на неё
были зелёные.

Разгадка в топологии баз: у проекта их две — **живая** (`DATABASE_URL`, в неё
ходит dev-бэкенд) и **тест-ветка Neon** (`DATABASE_URL_TEST`, на ней идут
db-тесты). План реализации сознательно требовал накатывать новую ревизию
только на тест-ветку (`just migrate-test`, «на прод не льём» до мержа).
Итог: тесты зелёные, потому что тест-ветка на `0003`; экран падает, потому что
живая база осталась на `0002`.

## Корень

Не баг кода — **отложенный деплой-шаг**. Функция из ревизии `0003` не была
накачена на ту базу, в которую смотрит dev-бэкенд.

## Фикс

```
cd backend; uv run alembic upgrade head   # 0002_compliance → 0003_category_sort_path
```

Ничего не коммитили — код был верный.

## Уроки

- **«Тесты зелёные» ≠ «работает на стенде»**, когда тесты и стенд смотрят в
  разные базы. У двухбазовой схемы это штатный режим, о нём надо помнить.
- Новая ревизия считается «доехавшей», только когда `alembic current` на
  **каждой** целевой базе показывает её. Проверка — одна команда.
- После мержа ветки с миграцией шаг `just migrate` на живой базе — обязательная
  часть выката, а не опция.
