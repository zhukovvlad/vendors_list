# 2026-07-11 — Карточка вендора (read-эндпоинты + мутации + матрица)

Ветка `feat/vendor-card`. Реализовано по методике subagent-driven-development
(план [docs/superpowers/plans/2026-07-11-vendor-card.md](../superpowers/plans/2026-07-11-vendor-card.md),
спека [.../specs/2026-07-11-vendor-card-design.md](../superpowers/specs/2026-07-11-vendor-card-design.md)):
9 задач (failing-first в TDD-тасках), свежий субагент-исполнитель на задачу +
двухстадийное ревью между тасками.

Цель — реализовать экран карточки вендора (что это за компания, где разрешена, какими
брендами владеет) с интерактивными мутациями (переключение соглашения, добавление/удаление
alias'ов) и кликабельность вендор-тегов в матрице каталога.

## Что сделано

### База данных

- **Миграция `0005_vendor_where_allowed`:** функция `vendor_where_allowed(vendor_id)` —
  единый источник правила исключения, которое определяет, где разрешён вендор по листингам.
  Правило: вендор разрешён на стандарте/класс-офиса, если существует хотя бы один листинг
  с этим вендором на открытом релизе (не-исключённый). Закрытый релиз исключается (boundary
  = release); DISTINCT ON результаты стабильно упорядочены по позиции категории.

### API

- **`GET /vendors/{id}`** — чтение карточки вендора (header): `id`, `name`, `kind`
  (локализуется на фронте), `note`, список `aliases` с паттерном маппинга на вендоров,
  `represents` (объект с `id`/`name` владельца или `null`), `represented_count` (счётчик вендоров,
  которых представляет этот), `starred` (из функции БД). Эндпоинт read-only, RBAC не требует.
- **`GET /vendors/{id}/where-allowed`** — дерево структуры разрешённости: стандарты → 
  классы-офисы → листинги позиций. Сервер пивотит `vendor_where_allowed()` на структуру
  вложенных объектов (чистый нesting предзагруженных рядов, нет N+1).
  API читает из функции БД напрямую.
- **`PUT /vendors/{id}/agreement`** (мутация toggle) — переключение флага соглашения для
  вендора. Асимметричное поведение O1 (no-op, если вендор уже в состоянии active без требуемого
  обновления; не реанимировать expired/terminated заявки). Транзакция через `Depends(tx)`,
  audit trigger срабатывает с корректным юзером через SET LOCAL. Возвращает `{"starred": bool}`.
  RBAC: admin only.
- **`POST /vendors/{id}/aliases`** (создание alias) и **`DELETE /vendors/{id}/aliases/{alias_id}`**
  (удаление) — глобально уникальные alias'ы, потребители могут создать alias на свой
  вендор (например, торговое имя = alias). В случае конфликта уникальности — 409.
  POST возвращает `{id, alias}`. RBAC: admin only.

### Фронтенд

- **`VendorCardScreen`** (`screens/vendors/`) — экран карточки с двумя блоками:
  - Шапка (имя/тип вендора, заметка, список alias'ов, блок brand-и-merge);
  - «Где разрешён» — аккордеон с раскрытием по стандарту, внутри классы-офисы и чипсы
    листингов (position, segment). Исключённые листинги зачёркнуты (инверсия цвета),
    легенда объясняет иконку/статус (разрешён, зачёркнут, нет-позиции и т.п.).
- **Интерактивный toggle соглашения** (switch-компонент) — статус из `useVendor`, toggle
  мутирует `useToggleAgreement`. На фронте отражает асимметрию: если уже active и toggle
  не требует UPDATE, то disabled-state или no-visual-feedback.
- **CRUD alias'ов** — кнопка добавить (диалог/форма) или иконка удалить на каждом alias.
  Хуки `useAddAlias`/`useRemoveAlias`, queryKey инвалидируется мутациями.
- **Кликабельность вендор-тегов в матрице** — в `MatrixScreen`, вендор-тег вместо обычного
  `<span>` стал `<Link to="/vendors/$vendorId">`. vendor_id уже был в payload от бэкенда
  (GET /listings/matrix), никаких изменений на бэкенде не потребовалось.
- **model.ts** — чистые хелперы локализации `kind` (type → RU-label), форматирование
  статусов блока «Где разрешён», склонение по числу (частичное, без полного
  pluralize-хелпера — отложено в TECH_DEBT).

### UI/DS

- **shadcn-примитивы:** `switch` (toggle соглашения) и `accordion` (разделы «Где разрешён»).
- **Новых цвет-токенов не потребовалось** — используются существующие `foreground`,
  `muted-foreground`, `destructive` (зачёркнутые листинги), `warning` (внимание).
- **a11y:** вендор-ссылка в матрице — `<Link>` (не `<button>`), accessible name из
  `vendor_name`.

## Ключевые решения

### Правило исключения — одна истина в SQL

Функция `vendor_where_allowed` — центральное место, в котором
определяется логика исключения. Никакой дублирование на фронте или в серверных фильтрах:
- API читает напрямую из функции (`SELECT * FROM vendor_where_allowed(...)`)
- Гарантирует консистентность: если кто-то напишет в БД минуя API, правило остаёт
  единственным источником истины
- DISTINCT ON упорядочивает результаты стабильно (по position) для reproducibility

### Два read-эндпоинта, чистый нesting

- **`GET /vendors/{id}`:** read-only метаданные + счётчик (из `COUNT` в SELECT,
  не отдельный запрос)
- **`GET /vendors/{id}/where-allowed`:** server-side pivot функции `vendor_where_allowed`
  в иерархию. Фронт получает готовое дерево, нет N+1 запросов

### Асимметрия toggle соглашения (O1) — почему no-op

Когда вендор уже в состоянии `active` (соглашение принято), а фронт отправит toggle
с тем же значением (попытка "включить" уже включённое):
- API не выполняет UPDATE (no-op)
- Audit trigger не срабатывает (в БД не изменилось)
- Нет "пустого" изменения в истории — чище

**Почему не реанимировать expired/terminated:**
- История соглашений — финализирована: раз expired/terminated, значит срок прошёл
  или договор расторгнут
- Реанимация требует бизнес-логики (переподпись, новый срок и т.п.) — вне API,
  должна быть отдельная мутация с явной семантикой
- Текущий toggle — простой `active/inactive` переключатель, не "история-воскрешение"

### Кликабельность матрицы — нулевой бэк-costs

Vendor_id уже был в payload матрицы (GET /listings/matrix), просто не был использован.
Изменение: вендор-спан стал `<Link to="/vendors/$vendorId">`. Нулевой новой логики на
бэкенде, чистый фронт-линк.

### Развилки O1–O4 (отложено)

- **O1 (done):** асимметрия toggle — implement.
- **O2:** диалог merge-flow (выбор целевого вендора, перенос листингов) — заглушка кнопки.
- **O3:** агрегация по brand-key на карточке (посмотреть брендов, которых представляет этот

- **O4:** тултип с label (release) вместо просто иконки на исключённых листингах — тултип

## Ловушки и уроки

### Prettier-debt из фронт-задач 4–6 (Task 8)

Задачи 4–6 (VendorCardScreen, мутации, кликабельность) выполнялись Haiku-субагентом,
который прогонял `vitest`/`tsc`, но не `format:check`. Prettier ошибки (trailing commas,
лишние скобки) прошли незаметно. Task 8 (финальное ревью фронта) поймал это через
`npm run format:check`, правки применены (нет новых коммитов, часть реализации 4–6).

**Урок:** `just ci` — единственный авторитет. Ad-hoc `npm run test`/`npm run typecheck`
недостаточны, пропускают `prettier --check`. Перед пушем — полный `just ci`, не бёрем
в голову отдельные проверки.

### Global-unique constraint на alias.pattern

`alias.pattern` должен быть глобально уникален (нельзя два разных вендора с одинаковым
alias). Реализовано как `UNIQUE(pattern)` на таблице `alias`. При конфликте —
`IntegrityError` (409 Conflict) на фронте. Трёх-стороннее соответствие: БД-схема,
API-ошибка, фронт-обработка.

## Отложено (см. TECH_DEBT)

- Агрегация «Где разрешён» по brand-key (O3): только листинги самого вендора на карточке,
  не полная картина всех брендов организации.
- Диалог merge-flow (O2): заглушка кнопки, без интерактивного выбора целевого вендора.
- Ленивое раскрытие «Где разрешён» по стандарту: одна загрузка всех листингов;
  сворачиваемое дерево с lazy-loading — после profile-анализа.
- RU-copy/плюрализация: фиксированные строки типа «1 позиций» (должно быть
  «1 позиция»); отложено до создания единого RU-pluralize хелпера.
