## 0. Резюме для ревьюера

Разовый admin-скрипт `python -m scripts.seed_vendors`, грузящий три стартовых
`.xlsx` в живые таблицы ядра (`category`, `position`, `vendor`, `agreement`,
`listing`) и, опционально, фиксирующий первое издание на каждый тип объекта
(`release` + `freeze_release`). Поддерживает `--dry-run`. Идемпотентен через
транзакционную полную перезагрузку стандартов с защитой от затирания
проектных данных.

**Что подтвердилось/изменилось после разведочного чтения трёх файлов**
(read-only скан, детали в §A):

1. **Маркеры требований — не те, что в примере ТЗ.** Реальные ячейки-требования
   в файлах — это ровно `Россия` и `По согласованию с Мосэнергосбыт`.
   `ГОСТ` как **самостоятельная** ячейка-требование **не встречается** — он
   всегда внутри названий продуктов («Трубы стальные ГОСТ 3262-75 …»). Счёт
   сходится с калибровкой: жилой 12×`Россия` + 6×`По согласованию…` = 18. §8.
2. **Классы в шапке совпадают с именами `segment` дословно** (после `.strip()`) —
   алиас-таблица не нужна. Файл жилого пишет `Делюкс (Элит)` (а не «Делюкс/Элит»,
   как в ТЗ). §11.
3. **Шапка не на фиксированной строке.** Жилой/соц — одноуровневая шапка на
   строке 5; офис — двухуровневая (группы стр. 6, классы стр. 7). Определять
   по якорю `№` в колонке A, а не по номеру строки. §4.2.
4. **В области данных есть объединённые ячейки, несущие данные** (жилой
   `C89:H93`): один набор вендоров растянут на 5 позиций × 6 классов. Без
   разворота merge теряются вендоры 4 позиций. **Разворот merge для колонок
   классов обязателен.** §4.4.
5. **Одиночная ячейка `Ujin`** (24× жилой) — это вендор «Ujin», а не флаг
   интеграции; правило «вырезать Ujin» нельзя применять, если имя схлопнется в
   пустое. §9.3.

**Решения §20 подписаны ревьюером 2026-07-09** (два дефолта изменены: §20.4
`--no-freeze`, §20.6 требование-в-`note`; новое требование к `--force` в §14).

## 1. Назначение и границы

**Делает:** разбирает 3 файла, раскладывает данные по живым таблицам ядра, не
пересчитывая ничего вычислимого (светофор/звезда/проценты — вьюхи/функции БД).

**Вход:** 3 `.xlsx` (по умолчанию из `temp/`, либо пути аргументами CLI).
**Выход:** записи в `category`, `position`, `vendor`, `agreement`, `listing`
(+ `change_log`/`agreement_change_log` через триггеры) и, опционально,
`release`/`release_listing` через `freeze_release`; печатный отчёт прогона.

**В сид НЕ входит** (подтверждаем границы ТЗ): нормализация имён/алиасы
(`vendor_alias`), `represents_id` (бренд-владелец), UI, экспорт, пересчёт
соответствия. Схема (`represents_id`, `vendor_alias`) позволит доработать это
позже без переимпорта.

## 2. Архитектура и слои (требование тестируемости)

Чистый разбор строго отделён от чтения xlsx и от записи в БД:

```
backend/app/seed/
    parse.py     # ЧИСТЫЕ функции: str/list -> dataclass. Без openpyxl, без БД.
    reader.py    # openpyxl: выбор листов, шапка, границы колонок, разворот merge,
                 # нормализация ячеек. Возвращает простые структуры, зовёт parse.
    loader.py    # SQLAlchemy Core (async): lookup справочников, вставка, транзакция,
                 # app.user, dry-run, отчёт, релизы. Зовёт reader+parse.
    report.py    # аккумулятор счётчиков/предупреждений + рендер отчёта.
backend/scripts/
    seed_vendors.py   # тонкий CLI-шим (argparse), зовёт app.seed.loader
```

Логика живёт в `app/seed/` (импортируема, покрывается `mypy app --strict` и
юнит-тестами без файлов/БД). CLI в `scripts/` — тонкий (как `scripts/export_openapi.py`).

> Замечание по mypy: `just typecheck` гонит `mypy app`, `scripts/` под него не
> попадает. Держим `scripts/seed_vendors.py` тривиальным (парс аргументов +
> `asyncio.run(loader.run(...))`), либо расширяем цель mypy на `scripts`
> (решение — §20.7).

**Ключевые dataclass чистого слоя** (черновик, финализируется на этапе плана):

```python
class RowKind(Enum): HEADING; POSITION; FOOTNOTE; BLANK      # BLANK — пустая строка
@dataclass RowClass:      kind; number: tuple[int,...]|None; name: str|None
@dataclass ParsedVendor:  name: str; starred: bool; ujin: bool; note: str|None
CellContent = MetaCell(status) | VendorList(list[ParsedVendor])
```

## 3. Порядок обработки файла (высокоуровнево)

Для каждого из 3 файлов:
1. Открыть `openpyxl.load_workbook(path, data_only=True)` (кэшированные значения
   формул; колонка A позиций = число, не `=A…`). §4.1.
2. Выбрать **видимый** лист (`sheet_state == 'visible'`). §4.2.
3. Найти строку-якорь шапки (`№` в A), определить одно/двухуровневую шапку,
   собрать классы→колонки и (для офиса) группы. §4.2–4.3.
4. Определить `building_type` по имени файла (residential/office/social). §11.
5. Идти по строкам данных: классифицировать (heading/position/footnote/blank),
   строить дерево категорий, для позиций читать ячейки классов (с разворотом
   merge), парсить содержимое. §5–§9.
6. Проверить целостность дерева (у каждого узла глубины >1 есть родитель). §6.
7. Сдать накопленные структуры в loader.

## 4. Слой чтения (reader.py)

### 4.1 Режим открытия
`data_only=True`. Обоснование: `position.source_ref` = вычисленный номер строки
как текст; колонка A позиций читается как `int`, что упрощает классификацию.
Проверено: кэш значений присутствует (A9 = `1`). Заголовки в A — литеральные
строки, режим на них не влияет.

### 4.2 Выбор листа и детекция шапки
- Берём **только** листы `sheet_state == 'visible'` (правило по флагу, не по
  чёрному списку имён). Скрытые (`Vendor list`, `Изделия`, `Материалы`,
  `Вариант 2`) — игнор, их имена печатаем в отчёт. Проверено: у каждого файла
  ровно один видимый лист (жилой — `Производители`; офис/соц — `Оборудование `
  с хвостовым пробелом).
- **Якорь шапки:** первая строка, где `strip(A) == '№'`. Пусть это строка `R`.
- **Одно/двухуровневая шапка:**
  - если в строке `R+1` пусты A и B, но есть непустые ячейки в колонках ≥3 →
    **двухуровневая** (офис): группы на `R`, классы на `R+1`, данные с `R+2`;
  - иначе → **одноуровневая** (жилой/соц): классы на `R`, данные с `R+1`.

### 4.3 Границы колонок классов (защита от фантомных колонок)
От колонки 3 вправо берём непрерывный ряд непустых ячеек строки классов до
первой пустой — это и есть колонки классов. **`max_column` не используем**
(в офисе он 16384). Проверено: жилой C–H (6), офис C–F (4), соц C (1).
Для офиса группа каждой правой колонки — ближайшая непустая ячейка строки групп
слева (объединённые: `Офисные здания` над Prime/Класс А/Класс B;
`Торгово-развлекательные центры` над ТРЦ).

### 4.4 Разворот объединённых ячеек (ОБЯЗАТЕЛЬНО для колонок классов)
Строим карту merge-диапазонов. При чтении ячейки класса (колонка ≥3) на строке
позиции: если ячейка покрыта merge — берём значение якоря (верхняя-левая).
Проверено: жилой `C89:H93` — один набор вендоров на 5 позиций × 6 классов; без
разворота 4 позиции теряют всех вендоров.
- Классификация строк (§5) использует **сырые** значения A и B (без разворота),
  т.к. горизонтальные merge заголовков `A…H` иначе протекли бы текстом в B.
  Горизонтальные merge встречаются только на строках-заголовках (не позициях),
  поэтому разворот колонок ≥3 на строках позиций безопасен.

### 4.5 Нормализация значения ячейки
- `str(value).strip()` (в части ячеек ведущий `\n`).
- Внутренние переносы строк в токене → пробел, сжать повторные пробелы
  (пример: `Трубы стальные ГОСТ 3262-75\n (…)`). Не трогаем содержимое иначе.
- Пустая (None/`''`) ячейка класса → **строка listing не создаётся** (пусто =
  `undefined`, мета-строку для пустого не заводим).

## 5. Классификация строк (parse.py, чистая)

По сырым A (`col_a`) и B (`col_b`):

| Тип | Условие | Действие |
|-----|---------|----------|
| **HEADING** | `col_a` матчит `^\s*\d+(?:\.\d+)*\.?\s+\S` **И** `col_b` пусто | узел дерева; глубина = число сегментов; имя = хвост после номера |
| **POSITION** | `col_a` — целое (или формула `=…`) **И** `col_b` непусто | позиция под текущим листом-заголовком; имя = `col_b` |
| **FOOTNOTE** | строка начинается с `Примечание: Ujin` | пропустить (счётчик в отчёт) |
| **BLANK** | A **и** B пусты | пропустить (трейлинг-заполнение Excel; не ошибка) |
| **иначе** | — | **ОШИБКА** с номером строки и содержимым; прогон падает |

`BLANK` — осознанное исключение из «всё непонятное → падать»: пустая строка
данных не несёт (max_row включает хвостовые пустышки). Строка с контентом в A,
но не распознанная как heading/position — это по-прежнему **ошибка** (защита от
будущих правок источника). Правило распознавания на всех 3 файлах отработало
без нераспознанных строк (см. калибровку §19).

## 6. Дерево категорий (`category`, общее для всех типов)

- **Идентичность узла = полный числовой путь** (напр. `1.1.1`). Первый файл,
  вводящий путь, задаёт имя узла; последующие файлы с тем же путём, но иным
  именем → **предупреждение** (конфликт имени), имя первого сохраняется.
  Обоснование: десятичная нумерация — это и есть явная структура, которую ведут
  авторы источника (ТЗ §2); ключ по номеру детерминирован и переживает
  расхождения регистра/пробелов в названиях (жилой «…Кондиционирование» vs
  соц «…кондиционирование»).
- `parent_id` = узел пути без последнего сегмента (`1.1.1` → `1.1`).
- `sort_order` = последний сегмент номера.
- **Целостность:** после сбора всех заголовков — у каждого узла глубины >1 есть
  родительский путь. Разрыв → ошибка (узел без родителя, с указанием пути/файла).
- **Расхождение «Благоустройство»** (не блокер): в жилом/соц — `2.4` (под
  материалами), в офисе — отдельный корень `4`. Разные пути → два узла. Это
  ожидаемо; приведение к единому виду — в источнике, не в скрипте.

> Альтернатива (идентичность по цепочке имён вместо номеров) — §20.2.

## 7. Позиции (`position`)

- **По строке на каждую POSITION-строку файла; между файлами НЕ дедуплицируем.**
  Обоснование: совпадает с калибровкой (жилой 266 / соц 235 / офис 242) и
  дословной формулировкой ТЗ §2 («строка position, привязанная к текущему
  листу-заголовку»). Одноимённая номенклатура в разных типах объектов — разные
  строки под общим узлом дерева; типовое разделение живёт в `segment`/`listing`.
- `category_id` = текущий лист-заголовок.
- `name` = `col_b`. `source_ref` = номер из колонки A (текст). `sort_order` =
  порядок в файле. `requirements` = **NULL** (на уровне позиции источника нет;
  скобочная область применения идёт в `listing.note`, см. §9.4).

## 8. Разбор содержимого ячейки (parse.py, чистая)

Порядок (ячейка уже `.strip()`-нута, §4.5):
1. Ячейка ∈ {`-`, `—`, `–`} → **одна мета-строка** `listing`:
   `status='not_applicable'`, `vendor_id=NULL`, `spec_text=NULL`.
2. Ячейка (нормализованная, регистронезависимо) — **требование**, если целиком
   совпадает с маркером из фиксированного набора:
   - точное `Россия`;
   - префикс `По согласованию` (покрывает `По согласованию с Мосэнергосбыт`).

   → **одна мета-строка**: `status='requirement'`, `spec_text=<исходный текст
   ячейки>`, `vendor_id=NULL`. Требование — на всю ячейку целиком.

   > `ГОСТ` намеренно **не** маркер (в файлах не встречается как самостоятельная
   > ячейка). Набор маркеров вынесен в константу — расширяется правкой кода, а
   > не источника.
3. Иначе → **список вендоров**. Сплит по запятой, не трогая запятые в скобках:
   `,(?![^(]*\))`. Каждый токен → §9. `sort_order` = позиция токена в ячейке.
4. **Смешанные ячейки** (напр. `Россия, Aquatherm green pipe sdr11*, sdr9(faser)`,
   встречается 6/4/1 раз): ячейку разобрать как **список вендоров** (ветка 3),
   а извлечённое ключевое слово требования (`Россия` / `По согласованию…` в
   начале ячейки, до первой запятой) — **не** делать вендором, а записать
   префиксом в `listing.note` **каждого** вендора этой ячейки: `Требование: <kw>`
   (напр. `Требование: Россия`). Плюс предупреждение в отчёте.
   Так смысл требования не теряется, а инвариант `listing_cell_chk` не нарушается
   (ячейка остаётся списком вендоров, без мета-строки). Если у вендора уже есть
   note из области применения (§9.4) — склеить через `; `
   (`Требование: Россия; паркинг`). Токен-требование в дальнейший разбор §9 не
   идёт (вендор из него не создаётся).

**Инвариант ячейки** (триггер `listing_cell_chk`): ячейка — ЛИБО вендоры, ЛИБО
одна мета-строка. Наши ветки 1–3 взаимоисключающи, инвариант соблюдается.

## 9. Разбор вендор-токена (parse.py, чистая)

Вход — один токен из §8.3. Порядок операций фиксирован:

### 9.1 Звезда `*` (соглашение)
`*` где угодно в токене (включая внутри скобок/кавычек) → `starred=True`; удалить
**все** `*` из имени. Звезда — глобальное свойство вендора (не ячейки):
агрегируется по всем вхождениям, агрумент создаётся один раз (§10).
Проверено: жилой 582 / офис 378 / соц 94 вхождения `*` (жилой сходится с ТЗ).

### 9.2 `Ujin`, приклеенный к бренду
Подстрока `Ujin` (обычный текст, не надстрочный — `font.vertAlign is None`).
Если после вырезания `Ujin` остаётся непустое имя → `ujin=True`, имя = остаток
(`MasterScada ERMUjin` → `MasterScada ERM`; `Яндекс.СтанцияUjin` → `Яндекс.Станция`;
`STRAZH (RUBEZH)*Ujin` → `STRAZH (RUBEZH)`, ujin=true, starred=true).

### 9.3 Одиночный `Ujin` (край, требует подтверждения — §20.1)
Если токен целиком `Ujin` (вырезание схлопнуло бы имя в пустое) → вендор
**«Ujin»**, `ujin=False` (это сам продукт/платформа, а не флаг интеграции).
Проверено: одиночный `Ujin` — 24/6/1 ячеек (жилой — 24, уточнено ревьюером;
на логику не влияет). Отчёт печатает их число.

### 9.4 Скобки
По умолчанию **оставляем в имени как есть** (`Midea (Bosch, Clivet)`,
`Hitachi (Hisense, Electrolux)` — целиком в `vendor.name`; запятая в скобках не
делит, §8.3).
**Исключение — область применения:** если содержимое первой скобки
(регистронезависимо) начинается с/содержит один из маркеров
`{для , паркинг, поквартирн, тех. помещ}` → имя = бренд до скобки (strip),
`listing.note` = содержимое скобки без внешних скобок. Проверено по ТЗ: жилой 4
уникальных (`АРКТИКА (для тех. помещений…)`, `Арктика (паркинг)`, `ТеплоМаш (…)`,
`ПУЛЬСАР (Поквартирные)`).

### 9.5 Кавычки
Внешние кавычки (`«»`, `"`, `„“`) в имени **не срезаем** (это нормализация —
отложена). `"ЭМ-Кабель"*` → имя `"ЭМ-Кабель"`, starred=true. Дедуп-протечки от
неоднородных кавычек допустимы на этапе сида (ТЗ §3.5).

## 10. Дедупликация вендоров и соглашения

- **`vendor.name` уникален.** Loader ведёт map `name -> vendor_id`; повторный
  токен переиспользует id. Разные написания (`System Air` vs `SystemAir`) →
  разные вендоры — **известно и допустимо** (ТЗ §3.5). Отчёт печатает число
  уникальных вендоров и список «похожих» имён (эвристика для человека, не
  действие).
- `vendor.kind = 'manufacturer'` (дефолт), `represents_id = NULL`, `note = NULL`.
- **Соглашения:** вендор считается «звёздным», если `starred=True` **хотя бы в
  одном** вхождении. Для каждого такого вендора — ровно одна строка `agreement`
  `status='active'` (даты/`doc_ref` NULL). Триггер `agreement_audit` пишет лог
  автоматически. Функция `vendor_starred()` затем вернёт true — звезду **не
  храним и не считаем** сами.

## 11. Маппинг классов файла → `segment`

`building_type`/`segment`/`segment_group` **уже засеяны** в `0001` (стр.432–466) —
сид их **не создаёт**, а делает **lookup по имени** в рамках типа объекта.
Тип объекта определяется по имени файла (подстроки `жилые`/`офисные`/`социальные`
→ `residential`/`office`/`social`).

Метка класса в шапке → `.strip()` → точное совпадение с `segment.name`. Проверено:
алиасы не нужны, метки совпадают дословно.

| Тип | Лист (видимый) | Строка классов | Колонки | Метки → `segment.name` |
|-----|----------------|----------------|---------|------------------------|
| residential | `Производители` | 5 | C–H | `Делюкс (Элит)`, `Премиум`, `Бизнес-Премиум`, `Бизнес`, `Комфорт`, `Эконом` |
| office | `Оборудование ` | 7 (группы 6) | C–F | `Prime`, `Класс А`, `Класс B` (группа `Офисные здания`), `ТРЦ` (группа файла `Торгово-развлекательные центры` → `segment_group.ТРЦ`) |
| social | `Оборудование ` | 5 | C | `Социальные объекты` |

- Любая метка класса, не совпавшая ни с одним `segment.name` типа → **ошибка** с
  перечнем доступных имён сегментов (защита от правок шапки).
- Группа офиса используется для мягкой сверки (сегмент уже несёт `group_id` из
  сида `0001`); имя группы файла `Торгово-развлекательные центры` ≠ `segment_group.ТРЦ`
  — сверяем по позиции колонки (какие классы под какой группой), не по имени;
  расхождение сетки → **громкое** предупреждение (сигнал, что шапка файла
  разошлась со схемой), но не блокер.

## 12. Вставка `listing`

На каждую (позиция × класс = сегмент) ячейку:
- мета-строка (`-` / требование): 1 строка `listing` (vendor_id NULL, spec_text
  по правилу §8);
- список вендоров: N строк `status='allowed'`, `vendor_id` дедуплицированного
  вендора, `sort_order` = порядок в ячейке, `ujin_integration`/`note` по §9;
- пустая ячейка: строк нет.

Триггеры ядра отрабатывают сами: `listing_stamp` (проставит `created_by =
current_app_user()` — §15), `listing_cell_chk` (инвариант), `listing_audit`
(журнал). Мы **не** ставим `created_by`/`updated_at` вручную.

Порядок вставки соблюдает FK: `category` (топологически, родители раньше) →
`position` → `vendor` → `agreement` → `listing`.

## 13. Издания (`release` / `freeze_release`)

**Решение ревьюера (§20.4): дефолт `--no-freeze`.** Сид грузит **только живое
состояние**; заморозку первого издания админ делает осознанно из приложения,
КОГДА стандарт вычищен (нормализация имён, `represents_id`, чистка смешанных
ячеек — всё это правится после импорта). Заморозка при сиде зафиксировала бы
`published`-снимок грязного первого издания — это неверная граница
ответственности (данные держит скрипт, процессом издания управляет человек).

- Механику `release` + `freeze_release` в коде **оставляем и тестируем** —
  меняется только дефолт флага. `--freeze` доступен тому, кто сознательно его
  запросит: тогда на тип объекта создаётся один `release` и вызывается
  `freeze_release(release_id, author)`.
- При `--freeze`: `label` = `«к Стандартам, 25.03.2026»` (дата из имени файла
  `2026-03-25`), `effective_date = 2026-03-25`. `freeze_release` сам делает
  снимок с денормализованными подписями и звездой на момент издания. После него
  релиз `published`, открытого релиза не остаётся.

## 14. Идемпотентность / перезапуск

**Модель: сид ВЛАДЕЕТ таблицами стандартов целиком; один прогон = полная
транзакционная перезагрузка.**

Внутри одной транзакции, до вставки:
1. **Защита:** если в `compliance.project` или `compliance.project_selection`
   есть строки → **отказ** (hard error): «стандарты уже используются проектами;
   правки — в приложении». Снятие только явным `--force` (опасно). Это и есть
   ответ на «как отличить сид-данные от ручных»: до появления проектов вся
   правда стандартов принадлежит сиду; после — источник истины приложение и сид
   не гоняют.
2. **Сброс — `DELETE` в FK-порядке** (решено на этапе плана, НЕ `TRUNCATE
   CASCADE`): `release_listing`, `release`, `listing`, `change_log`,
   `agreement`, `agreement_change_log`, `position`, `vendor_alias`, `vendor`,
   `category`. `building_type`/`segment`/`segment_group` **не трогаем** (сид
   схемы). serial-id при `DELETE` не сбрасываются — по §14 не контракт.
   Аудируемую таблицу удаляем ПЕРЕД её журналом (`listing`→`change_log`,
   `agreement`→`agreement_change_log`): её `DELETE` пишет 'delete'-строки в
   журнал через AFTER-DELETE-триггер, и следующий `DELETE` журнала их стирает —
   иначе (журнал раньше родителя) на каждый ре-сид копились бы осиротевшие
   строки в `agreement_change_log` (у него нет FK на `agreement`).

   **Требование ревьюера к `--force` (жёсткое) — выполнено конструкцией:**
   `--force` снимает только проверку п.1, но **НИ ПРИ КАКИХ УСЛОВИЯХ** сброс не
   трогает `compliance.project` / `project_selection`. `DELETE` (без `CASCADE`)
   этого и не может: FK `RESTRICT` из `project.release_id` /
   `project_selection.(vendor_id|position_id)` **физически заблокирует** удаление
   строк ядра, на которые ссылаются проекты — `DELETE` упадёт, транзакция
   откатится, проекты целы. Т.е. снести стандарты «из-под» активных проектов
   нельзя даже с `--force` (свойство сильнее guard). Тесты — §18.
   `--force`.
3. **Загрузка** заново.

Одна транзакция на весь прогон: любой сбой → полный откат, БД чистая.
Повторный прогон (до появления проектов) даёт то же логическое состояние
(идентификаторы-serial не являются контрактом). **Альтернатива** (upsert по
естественным ключам) отклонена: у `category`/`position`/`listing` нет
естественного уникального ключа, реконсиляция дерева/мягких удалений сложна и
недетерминирована. Развёрнуто — §20.3.

## 15. Атрибуция авторства (аудит)

Транзакция **первым запросом** ставит `app.user` (как `db.tx`, ТЗ §золотое
правило 3):
```sql
SELECT set_config('app.user', :login, true)   -- SET LOCAL-семантика, bind-параметр
```
`:login` = `--author` (дефолт **`seed`**). Тогда `current_app_user()` вернёт
`seed`, и все `created_by`/`changed_by` в `listing`/`change_log`/`agreement*`
подпишутся `seed`. Логин — bind-параметром, не конкатенацией.

Движок берём из `app.db.get_engine()` (тот же `DATABASE_URL`, `statement_cache_size=0`
для PgBouncer). Открытие: `async with get_engine().begin() as conn:` (коммит/откат
как в `tx`).

## 16. Dry-run и отчёт прогона

- **`--dry-run`:** весь разбор выполняется, БД **не пишется** (транзакция
  открывается, выполняется, затем `ROLLBACK`; либо write-путь не вызывается).
  Печатается тот же отчёт, что и при реальном прогоне.
- **Отчёт** (на файл и суммарно):
  - листы: использован видимый `<имя>`; пропущены скрытые `<имена>`;
  - строки: заголовков, позиций, сносок (Ujin), пустых пропущено;
  - ячейки: непустых, `-`, требований, списков вендоров, всего токенов;
  - вендоры: уникальных, соглашений (звёзд) создано, вхождений `*`;
  - listing: строк к вставке в разбивке по `status`;
  - категории: узлов в объединённом дереве;
  - релизы: какие зафиксированы (если `--freeze`);
  - **предупреждения** (список, distinct): смешанные ячейки → требование-в-note
    (§8.4), одиночные `Ujin` (§9.3), конфликты имён категорий (§6), расхождение
    сетки групп офиса (§11), извлечённые note-области (§9.4), имена в кавычках
    (§9.5), «похожие» имена вендоров (§10).
- **Ошибки** — с локацией `<файл>:<лист>:<строка>[:<колонка>]` и содержимым;
  прогон падает (ненулевой exit code), БД не изменяется.
