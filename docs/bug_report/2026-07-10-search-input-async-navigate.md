# Поиск «зависает» при быстрой печати: контролируемый инпут от асинхронного роутера

**Дата:** 2026-07-10 · **Ветка:** `feat/listing-matrix` (PR #12) · **Серьёзность:** регресс UX (поле нерабочее)
**Файлы:** `frontend/src/screens/matrix/MatrixScreen.tsx`
**Внесён:** `3b4dcd6` (фикс замечания ревью) · **Фикс-коммит:** `61888a8`

## Симптомы

Поле поиска матрицы клинило при обычной скорости печати: символы «отскакивали»,
поле переставало реагировать, страница ощущалась зависшей. В консоли — пусто,
ошибок нет. До правок по замечаниям ревью бага не было.

## Расследование

- Backend чист: в логах запросы 200, без штормов.
- jsdom-тесты проходили — баг **не воспроизводился в тестовой среде** вообще.
- Реальный браузер (Playwright + Chromium): медленный ввод (1 символ в 1.2с)
  работает идеально; быстрый ввод — поле никогда не «стабилизируется», драйвер
  таймаутит. Значение отскакивает между нажатиями.
- Регресс-окно указал не git bisect, а простое сопоставление: баг появился
  ровно после коммита, где по замечанию ревьюера инпут перевели с
  `defaultValue` (неконтролируемый) на `value={search.q}` (контролируемый).

## Корень

Связка «контролируемый инпут ← напрямую асинхронный источник истины»:

1. Ревьюер справедливо заметил: `defaultValue` не синхронизируется с URL при
   back/forward. Инпут сделали контролируемым от `search.q`.
2. Но `navigate()` у TanStack Router **асинхронный**: между нажатием клавиши и
   обновлением `search.q` есть зазор.
3. При быстрой печати React на каждый рендер сбрасывает value поля на
   «отставшее» `search.q` → введённые символы пропадают/прыгают → поле
   фактически неуправляемо. Ошибки нет — потому и консоль пустая.

## Фикс

Стандартный паттерн «локальное состояние + дебаунс + обратная синхронизация»:

- инпут контролируется **локальным `useState`** — обновление синхронное,
  значение не отскакивает;
- в URL пишем **дебаунсом 300мс** (заодно 1 запрос вместо запроса на каждый
  символ — закрыло и отдельное замечание ревью про дебаунс);
- обратная синхронизация из URL (back/forward) — отдельным эффектом с гвардом
  `qInput === search.q`, чтобы два эффекта не зациклились.

Плюс регресс-тест: быстрый ввод не отскакивает, `q` попадает в URL после
дебаунса.

## Уроки

- **Контролируемый инпут нельзя вешать напрямую на асинхронный источник
  истины** (URL, роутер, сервер). Между ними обязан стоять синхронный слой —
  локальное состояние.
- **Фикс по замечанию ревью — тоже изменение поведения** и нуждается в проверке
  поведением, а не только «замечание закрыто». Ревьюер был прав про баг
  синхронизации, но буквальная реализация его предложения внесла регресс хуже
  исходного.
- Баги «работает медленно — ломается быстро» почти всегда про гонку состояний;
  jsdom их не ловит (события там не имеют реальной временной плотности) —
  нужен реальный браузер.
