# 2026-07-10 — Генезис-издание из сида (3 published per building_type)

Ветка `feat/genesis-release`. Спека
[.../specs/2026-07-10-genesis-release-design.md](../superpowers/specs/2026-07-10-genesis-release-design.md)
(бриф-исток — transient working-note `genesis-release-KICKOFF.md`, удалён после
исполнения). Работа выполнена по методике brainstorming → spec → раннбук (без
формального плана: продакшн-кода нет, «план» ≡ операционный раннбук).

Цель — превратить вчера засеянный live-перечень (13 244 строки в `listing`, без
единого `release`) в **3 базовых `published`-издания** — по одному на building_type.
До генезиса дашборд показывал Позиции = 0, Издания = 0 при заполненной БД.

## Ключевая находка: механику писать не пришлось

Бриф формулировал форки 2/3/4 как «решить, где живёт логика фиксации / аудит /
label». При старте выяснилось, что фиксация **уже реализована и покрыта тестами**:

- **Флаг `--freeze` у сида** ([loader.py:325-344](../../backend/app/seed/loader.py#L325-L344)):
  на каждый присутствующий building_type создаёт `release(status='open')` с
  `label='к Стандартам, {дата из имени файла}'` и `effective_date`, затем зовёт
  `freeze_release`. Аудит: `execute()` ставит `set_config('app.user','seed',local)`
  до записи, `freeze_release` получает `author='seed'`.
- **API `POST /releases/{id}/freeze`** ([releases.py:51](../../backend/app/routers/releases.py#L51)),
  `GET /releases`, `GET /releases/{id}/listing` — admin, через `tx`.
- **Функция БД `freeze_release`** ([0001_core_schema.sql:360-402](../../backend/migrations/sql/0001_core_schema.sql#L360-L402)).

Следствие: генезис — **операционный прогон уже протестированного пути**, ноль
нового продакшн-кода (миграции/API/парсер не тронуты). Способ фиксации —
пере-прогон `just seed --yes --freeze` (reset+reseed+freeze). Он оправдан именно
тем, что эндпоинта создания издания (`POST /releases`) в системе пока нет — сид-флаг
единственный путь создать `release`-строку. Это следствие текущих границ (§5/§6
не построены), а не костыль.

## Решения по форкам (утверждены заказчиком)

- **Форк 1 (артефакты имён):** фиксируем **как есть**. Парсер сознательно не
  срезает кавычки («нормализация в источнике, не в скрипте» — devlog сида). Генезис —
  базовая точка; неизменяемость снимка не пугает, правки пойдут через источник /
  будущий админ-контур с перевыпуском.
- **Форк 5 (open-черновик после):** **не создаём**. `seed --freeze` оставляет
  3 published + 0 open. Новый open появится при старте редактирования (§6).
- **Форк 6 (идемпотентность):** через `_reset` — повторный прогон сперва DELETE-ит
  live-таблицы, поэтому всегда ровно 3 издания, без дублей.
- **Бэкап-ветка Neon:** не делали. Решение заказчика: dev-окружение, данные
  воспроизводимы из Excel, «не плодить сущности».

## Раннбук (что фактически сделано)

1. **Pre-flight** `just seed-verify` (dry-run + калибровка §19, без БД): парсер
   сошёлся — allowed 7506+4509+1099 + n/a 97 + requirement 33 = **13 244**.
2. **Grounding (before)**, живым read-only запросом (не по памяти): `release=0`,
   `release_listing=0`, live `listing=13 244` (soft-delete-ов 0), dashboard
   positions=0/releases=0/drafts=0, вендоры 898/117. **Отпечаток last-write:**
   у `listing` `min(updated_at) == max(updated_at) == 2026-07-09 16:17:53Z`,
   `updated_by = {seed}` — доказательство, что ручных правок после сида не было
   и деструктивный reset ничего не сотрёт молча (guard `_guard_no_projects`
   проверяет только `compliance.project*`, ручные правки live-слоя он не ловит).
3. **Пауза на явный go-ahead** заказчика (шаг необратим на месте).
4. **Execute** `just seed --yes --freeze` против боевого Neon: `_reset` →
   reseed из 3 Excel → 3× `freeze_release`. Завершилось «✅ Записано в БД».
5. **Post-verify (все инварианты зелёные):**

| Инвариант | Результат |
|---|---|
| `release` | 3 строки, все `published`, `frozen_at` заполнен, `author='seed'`, `effective_date=2026-03-25`, по одному на building_type |
| `label` | `«к Стандартам, 2026-03-25»` у всех трёх |
| Строгое равенство | `SUM(release_listing)=13 244 == COUNT(listing WHERE deleted_at IS NULL)=13 244` |
| Снимки per type | social 1109 · office 4545 · residential 7590 (сумма 13 244) |
| `dashboard_summary` | positions_active **737**, releases_published **3**, drafts_open **0**, vendors 898 / with_agreement 117 |

Снимки per type точно бьются с калибровкой парсера (residential 7506+66+18=7590,
office 4509+24+12=4545, social 1099+7+3=1109) — сквозная сходимость Excel → live →
снимок.

## Строгое равенство вместо «сопоставимо» (правка по ревью)

Первая редакция спеки формулировала post-verify мягко («суммарно сопоставимо с
live listing»). По замечанию ревьюера заменено на **точный инвариант**: `freeze_release`
копирует все живые строки типа без фильтра по статусу
(`WHERE seg.building_type_id = v_bt AND l.deleted_at IS NULL`, INNER JOIN по
`position`/`segment` — оба `NOT NULL`), каждая живая строка принадлежит ровно одному
building_type → `SUM == COUNT(live)`. Это ловит реальный класс ошибок (осиротевший
building_type сегмента, неучтённый soft-delete). Инвариант выполнился точно.

## Явные факты на будущее

- **`vendor.id` (и прочие serial) НЕ стабильны между прогонами сида.** Сброс —
  `DELETE`, не `TRUNCATE RESTART IDENTITY` (сознательное решение из декомпозиции
  write-path, чтобы не задеть `compliance.project*`), поэтому id после reseed
  поехали вверх. Сейчас безопасно — на них никто не ссылается (0 проектов, 0
  прежних изданий). Если где-то появится захардкоженный `vendor.id` — он переживёт
  reseed только до первого проекта. С появлением проектов пере-прогон сида уже
  недопустим (guard `_guard_no_projects` заблокирует без `--force`).
- **`vendor` не имеет audit-колонок** (`updated_at`/`updated_by`) — правки имён
  таблицей не аудируются; их аудит-контур — `agreement_change_log` по соглашениям.
  Поэтому отпечаток last-write снимали по `listing` (главная редактируемая
  поверхность с аудитом).
