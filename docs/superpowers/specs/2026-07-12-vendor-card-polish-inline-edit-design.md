# Карточка вендора: визуальный редизайн + инлайн-правка шапки — Design

**Дата:** 2026-07-12
**Статус:** на ревью заказчика
**Ветка (планируется):** `feat/vendor-card-polish`

## Цель

Один срез = **новый визуальный дизайн карточки вендора** (утверждён заказчиком, референс-макет
приложен) + **инлайн-правка шапки** (имя и примечание) поверх новой вёрстки. Редизайн и правку
НЕ разделяем по шапке: сначала новая вёрстка, поверх — редактирование.

Предыдущий срез (карточка вендора, PR #20, merged) дал функционал; этот — доводит визуал до
утверждённого дизайна и добавляет редактирование идентичности.

## Объём

**В срезе:**
- Редизайн всех блоков карточки в карточки-поверхности (`bg-card`) по макету.
- Бэкенд: `PATCH /vendors/{id}` (имя+примечание, partial-семантика, rename→alias, коллизии→409).
- Фронт: инлайн-правка имени (в `<h1>`) и примечания (Notion/Linear-паттерн).

**Follow-up (отдельные срезы, НЕ здесь — но фиксируем, чтобы не потерять):**
- **C1 — чистка имён после сида:** ~47 имён с висящими кавычками (`"Русский бойлер`) и
  хвостами `(Native)`/`(Нативный)`. Bulk-нормализация: каноническое имя чистим, грязное
  написание → `vendor_alias`. Переиспользует механику B1 (rename→alias). Отдельный срез:
  one-off скрипт/ревизия с ручной сверкой (человек подтверждает маппинг — ловушка ТЗ §3.4).
- **C2 — сегменты соцобъектов:** тип «Социальные объекты» имеет 1 сегмент с именем самого типа
  (заглушка сида; Жилые=6, Офисные=4, Соц=1). Чип класса с именем типа выглядит багом. Корень —
  данные/продуктовое решение о сегментации соцобъектов. В ЭТОМ срезе рисуем как есть; решение
  отложено до продуктовой развилки (в TECH_DEBT).

**Вне объёма:** боевой SSO/RBAC (§2 — плейсхолдер), правка `kind`/бренда/соглашения (соглашение
уже тумблер), поток «Объединить» (O2, заглушка).

## Золотые правила (наследуются)

Schema-first без ORM (SQLAlchemy Core `text()`); пишущие эндпоинты только через `Depends(tx)` +
`require_admin` (viewer→403); UI на русском; **только семантические токены DS, ничего нового не
заводить**; `main` зелёный (`just ci`); после бэкенд-контракта — `just types`.

---

> **Метки пр.1–пр.4** (в §B/§C) — правки заказчика к инлайн-правке: **пр.1** — при
> переименовании старое имя уходит в `vendor_alias`; **пр.2** — коллизия нового имени с чужим
> `vendor_alias.alias` → 409; **пр.3** — очистка `note` (два теста: `""`→`NULL` и «поле не
> пришло»); **пр.4** — кнопка-дисплей и инпут живут внутри `<h1>` (не теряем `role=heading`).

## A. Редизайн — сопоставление токенов и вёрстка

Макет — тёмная тема на сырых hex. Используем **только семантические токены** (тема
переключается автоматически, светлая «просто работает»). Сопоставление (hex тёмной темы →
токен, сверено по [index.css](../../../frontend/src/index.css)):

| Роль в макете | hex (dark) | Токен | Tailwind |
|---|---|---|---|
| Фон карточки-блока | `#16121F` | `--card` | `bg-card` |
| Рамка | `#2A2640` | `--border` | `border-border` |
| Elevated-подложка (аватар, бейдж kind, чипы классов) | `#1E1930` | `--accent` | `bg-accent` |
| Sunken-подложка (легенда) | `#110D1C` | `--muted` | `bg-muted` |
| Пунктир (dashed-чипы) | `#3A3556` | `--border-strong` | `border-border-strong border-dashed` |
| Акцент (инициал, иконки, «+вариант») | `#8B66F0` | `--primary` | `text-primary` |
| Вторичный текст | `#B5B2C4` | `--muted-foreground` | `text-muted-foreground` |
| Основной текст | `#F4F2FA` | `--foreground` | `text-foreground` |
| Радиус карточек/аватара | 10px | `--radius-xl` | `rounded-xl` |
| Радиус чипов/бейджей | 4px | `--radius-md` | `rounded-md` |
| Пилюля (kind, «+вариант») | 999px | — | `rounded-full` / `rounded-md` |

**Осознанные упрощения (token-purity, «ничего нового не заводить»):**
- Макет использует 5-ступенчатую шкалу серого (`#76728C`, `#534F68` без токенов) — схлопываем
  в `text-foreground` / `text-muted-foreground`. Иерархия чуть площе макета; новый серый токен
  НЕ заводим (при желании заказчика — отдельная правка DS).
- Акцент в макете — `#8B66F0` (это `--primary-hover` тёмной темы); используем базовый
  `text-primary` (`#754AE8`). Небольшая разница насыщенности, токен-чистота важнее.
- Делитель строк аккордеона в макете `#1C1830` (нет токена) → `border-border`. Радиус легенды
  6px (нет токена) → `rounded-md` (4px).

**Иконки:** макет на Tabler, проект — `lucide-react`. Маппинг: `award`→`Award`, `plus`→`Plus`,
`arrows-join`→`Merge`, `chevron-*` — штатные у `accordion`, звезда соглашения — `Star`.

### Раскладка блоков

Контент — колонка `max-w-[720px] mx-auto`, зазор `space-y-3` (12px). Каждый блок —
`bg-card border border-border rounded-xl`. Никаких блоков «плашмя на фоне».

1. **Шапка** (`p-[18px_20px]`): ряд `flex items-center gap-3.5`:
   - Аватар-плитка 44×44 `rounded-xl bg-accent border border-border`, по центру — первая буква
     имени, `text-primary` 19px medium. (До C1 у грязных имён инициал может быть `"` — корень
     чинит C1.)
   - Центр (`flex-1 min-w-0`): ряд «имя (h1, 20px medium, `tracking-tight`) + пилюля `kind`
     (`bg-accent border-border text-muted-foreground rounded-full`) + бейдж `⭐ соглашение`
     (только при `starred`)»; ниже (`mt-1.5`) строка бренда — иконка `Award` 13px +
     «самостоятельный бренд» / «представляет: X» (`text-xs text-muted-foreground`); ниже —
     **строка примечания** (см. §B, инлайн-правка).
   - Справа: «Соглашение» + `Switch`. Выключенный тумблер нейтральный (`bg-border`, бегунок
     `bg-muted-foreground`), не акцентный; включённый — `bg-primary`.
2. **Варианты написания** (`p-[15px_20px]`): label uppercase `text-muted-foreground`. Пусто →
   «вариантов пока нет» (`text-muted-foreground`) + пунктирный чип `+ вариант`
   (`border-border-strong border-dashed text-primary`). Есть → чипы (`bg-accent border-border
   rounded-md`) с кнопкой удаления + пунктирный `+ вариант`. Сиротливое «—» убрать.
3. **Бренд и объединение** (`p-[15px_20px]`): иконка `Award` `text-primary` + «Самостоятельный
   бренд / не представляет другого вендора» (или «Представляет: X» ссылкой) + при
   `represented_count>0` строка «представляет N брендов» (со склонением). Справа —
   `disabled`-кнопка «Объединить · скоро» (`border-border`, иконка `Merge`, «· скоро»
   мелким `text-muted-foreground`).
4. **Где разрешён** (`p-[15px_20px]`): label + справа сводка «N стандартов · M позиций»
   (склонение обоих). Строки-стандарты — аккордеон с волосяными делителями
   (`border-t border-border`), шеврон `text-primary` открытый / `text-muted-foreground`
   закрытый, имя типа (14px medium) + «N позиций» справа. Раскрытый: `ml-6 border-l
   border-border pl-4` — имя позиции (13px) + чипы классов (`bg-accent border-border rounded-md`;
   excluded — `border-border-strong border-dashed line-through text-muted-foreground` + тултип с
   label релиза, **как уже реализовано**). Легенда внизу — в `bg-muted border border-border
   rounded-md`, клауза про зачёркивание — только при наличии excluded (**как уже реализовано**).

Дизайн тема-агностичен: используем токены, светлая тема рендерится корректно без отдельной вёрстки.

## B. Бэкенд — `PATCH /vendors/{id}`

**Схема** `VendorHeaderUpdate`: `name: str | None = None`, `note: str | None = None`.
`field_validator("name")` тримит; при непустом-но-после-трима-пустом → ошибка валидации (422).
В эндпоинте `data = body.model_dump(exclude_unset=True)` — различаем «поле не пришло» vs
«`note: null` (очистить)».

**Эндпоинт** `PATCH /vendors/{vendor_id}` → `VendorCard`, `Depends(tx)` + `require_admin`:
1. `_ensure_vendor` (заодно берём текущее `name` как `old_name`).
2. Если `"name" in data` и `new_name != old_name`:
   - **Коллизия (пр.2):** `SELECT 1 FROM vendor_alias WHERE alias=:new AND vendor_id<>:id` →
     если есть → **409** «Имя уже занято» (UNIQUE на `vendor.name` ловит только имя-имя;
     имя-vs-чужой-alias — нет, проверяем явно).
   - `DELETE FROM vendor_alias WHERE vendor_id=:id AND alias=:new_name` — если ново́е имя было
     собственным алиасом, снимаем дубль «имя == свой алиас».
   - `UPDATE vendor SET name=:new_name WHERE id=:id` — при коллизии имя-имя asyncpg даст
     `IntegrityError` → ловим → **409** «Имя уже занято».
   - **rename→alias (пр.1):** `INSERT INTO vendor_alias(vendor_id, alias) VALUES(:id, :old_name)
     ON CONFLICT (alias) DO NOTHING` — старое написание сохраняем как алиас (по нему ищут/
     импортируют). `ON CONFLICT DO NOTHING` делает переименование туда-обратно идемпотентным
     (A→B→A не падает на UNIQUE).
3. Если `"note" in data`: тримим; пусто → `NULL`; `UPDATE vendor SET note=:note WHERE id=:id`.
4. Возврат: `_load_vendor_card(conn, vendor_id)`.

**Рефактор:** сборку `VendorCard` из `get_vendor` вынести в хелпер `_load_vendor_card(conn, id)`
и переиспользовать в `GET` и `PATCH` (без дублирования).

## C. Фронт — инлайн-правка

**Компонент `InlineEditText`** (изолированный, юнит-тестируемый).
Пропсы: `value`, `onSubmit: (next) => Promise<void>`, `multiline?`, `ariaLabel`, `placeholder?`,
`error?: string | null`, классы дисплея/инпута.
Поведение:
- Дисплей — фокусируемая `<button>` (клавиатура + a11y, `aria-label`); текст стилизован под
  контент. Клик / Enter / Space → режим правки.
- Правка — `<input>` (single) / `<textarea>` (multiline), `autoFocus`, `value=draft`.
  Single: Enter → submit, Esc → отмена. Multiline: Esc → отмена, Enter — перенос строки. Оба:
  blur → submit.
- submit: `draft.trim() === value` → выйти без запроса (no-op); single и `draft.trim() === ""`
  → отмена (revert; имя обязательно); иначе `await onSubmit(draft)` — **reject → остаёмся в
  правке** (родитель через `error`-проп рисует ошибку), resolve → выходим.
- `error` (например, 409 «Имя уже занято») рендерится под полем (`role="alert"`,
  `aria-describedby`). При новом входе в правку ошибка очищается.

**h1-семантика (пр.4):** имя — заголовок экрана; кнопка-дисплей и инпут живут ВНУТРИ `<h1>`
(`<h1><InlineEditText/></h1>`), не вместо него — `role=heading` не теряем. Инпут раскрывается на
месте текста, не ломая ряд «аватар + имя + бейджи + тумблер».

**Хук `useUpdateVendorHeader(id)`:** `PATCH`; `onSuccess` инвалидирует `["vendor", id]` +
`["matrix"]` + `["dashboard"]` (имя видно в матрице/дашборде). Экран зовёт `mutateAsync` и мапит
409 → строка ошибки для соответствующего поля.

**Примечание (note):** редактируемая строка в шапке под строкой бренда (`text-xs
text-muted-foreground`). Пусто → кликабельный плейсхолдер «+ примечание» (subtle). Клик →
`textarea` на месте. При стирании текста фронт шлёт **пустую строку/`null`** (не опускает поле),
чтобы бэкенд очистил в `NULL` (пр.3).

## Тесты

**Backend (`test_vendors.py`, маркер `db`):**
- PATCH имени: 200, имя изменено, возвращён `VendorCard`; **старое имя добавлено в алиасы** (пр.1).
- Переименование туда-обратно (A→B→A) не падает на UNIQUE (пр.1, `ON CONFLICT`) **и** конечное
  состояние алиасов — ровно `{B}` (name=A, один алиас B, мусор не накопился): проверяет связку
  «DELETE своего алиаса, совпавшего с новым именем → INSERT старого имени», а не только «не упало».
- PATCH имени = чужое `vendor.name` → 409 (`IntegrityError`).
- PATCH имени = чужой `vendor_alias.alias` → 409 (пр.2, явная проверка).
- PATCH `note: ""` → `NULL` (пр.3); PATCH без `note` в теле → примечание не тронуто (пр.3,
  два раздельных теста, `exclude_unset`).
- PATCH пустого/пробельного имени → 422.
- PATCH несуществующего вендора → 404; PATCH под `viewer` → 403.

**Frontend:**
- Юнит `InlineEditText`: клик → инпут; Enter (single) вызывает `onSubmit`; Esc → откат к `value`;
  blur → submit; no-op (draft==value) не вызывает `onSubmit`; single + пусто → откат;
  reject `onSubmit` → остаёмся в правке; `error` рендерится.
- Экран: клик по имени → инпут в `<h1>`; ввод + Enter → `PATCH {name}`; 409 → инлайн-ошибка,
  остаёмся в правке; правка примечания → `PATCH {note}`; стирание note → тело с пустым `note`.
- Редизайн: смоук на структуру (карточки-поверхности, пустое состояние алиасов «вариантов пока
  нет», сводка «N стандартов · M позиций», склонения) — без хрупких проверок пикселей.
