# 2026-07-13 — Режим правки карточки вендора + типографика v3 «Где разрешён»

Ветка `feat/vendor-card-edit-mode` от чистого main (предшественники PR #20/#21/#22 влиты).
Реализовано по методике **subagent-driven-development**: 7 задач, свежий implementer-субагент
на задачу + целевое ревью (spec + quality) после каждой, финальное whole-branch ревью в конце.
Ревью Task 2 (scoped-WHERE исключения + трансляция `listing_cell_chk`→409) и Task 5 (самый
большой диф) — на Opus; остальные implementer/reviewer — Sonnet; оркестрация — Opus.

Источник: [спека](../superpowers/specs/2026-07-13-vendor-card-edit-mode-design.md),
[план](../superpowers/plans/2026-07-13-vendor-card-edit-mode.md).

Цель — отделить чтение от правки на карточке вендора (view↔edit), дать редактирование
разрешений (исключить/добавить вендора по классу/позиции/стандарту), довести блок
«Где разрешён» до типографики v3.

## Несущая ось: «черновик — не песочница»

Ключевое архитектурное решение (закрыто заказчиком до реализации): **правки идут ПРЯМО в
живой `listing`**. У таблицы `listing` нет `release_id` — она и есть текущее состояние.
Открытый `release` (`status='open'`) — лишь **маркер** «есть неопубликованные изменения»
(его считает дашборд как черновик). Кнопка «Готово» в режиме правки **ничего не коммитит** —
каждая операция уже применена к живым данным в момент клика. Обратимость держится на
операции «вернуть» (un-delete-first-else-INSERT) и неизменяемости published-релизов.

Отсюда весь бэкенд-контур: мутация пишет в `listing`, затем (и только при реальном
изменении) обеспечивает существование open-маркера через `ensure_open_release`.

## Что сделано (по задачам)

### Task 1 — ревизия `0006_ensure_open_release` (коммит `57c7ee2`)

SQL-функция `ensure_open_release(p_bt int) RETURNS int` — обеспечивает ЕДИНСТВЕННЫЙ
open-маркер релиза на тип объекта (симметрия с `freeze_release`; инвариант в БД,
golden-rule #6). Гонку снимает **ограничение** `uq_release_one_open ON release
(building_type_id) WHERE status='open'` (0001), не сервисный лок: параллельные вызовы
конфликтуют по частичному уникальному индексу через `ON CONFLICT (building_type_id)
WHERE status='open' DO NOTHING`, проигравший добирает чужой id вторым `SELECT`
(в plpgsql `INTO` без STRICT на 0 строк — не ошибка, `v_id` остаётся `NULL`).

`label` — **нейтральный и не ложь**: `freeze_release` НЕ перезаписывает `label`
(0001:396-400), значит любой авто-label доехал бы до published-релиза как есть.
«`<тип> — рабочая версия`» ничего ложного не утверждает; человек уточнит его при
публикации (экран изданий, §5). Обычный TDD (логика новая): db-тест RED до наката 0006
на тест-ветку, GREEN после. Ревизия накачена и на тест-ветку (`just migrate-test`), и на
боевую (`just migrate`) — head=`0006` подтверждён (иначе живой app 500-ит на первой правке).

### Task 2 — listing-мутации + `PATCH kind` + `/meta/positions` (коммиты `46c74e4`, `a439f05`)

Три эндпоинта + вспомогательная механика:
- `POST /vendors/{id}/listings` `{position_id, segment_ids[]}` → 204. Общий для «+ класс»/
  «+ позиция»/«+ стандарт». `_add_one_class`: живой → no-op; иначе un-delete последней
  soft-deleted строки (чистим и `deleted_by`), иначе INSERT allowed.
- `POST /vendors/{id}/listings/exclude` `{scope, position_id?, segment_id?, building_type_id?}`
  → `{excluded_positions, excluded_classes}`. **ОДИН scoped-эндпоинт** (решение заказчика,
  не три ручки): масштаб отдаётся из одного места, tx-обвязка не троится, будущая
  редактируемая матрица предпочтёт один вызов. Обязательность полей по scope — pydantic
  `model_validator(mode="after")`.
- `POST /vendors/{id}/listings/restore` `{position_id, segment_id}` → 204. Переиспользует
  `_add_one_class` (одна семантика «оживить класс»).
- `PATCH /vendors/{id}` +`kind` (валидатор enum → 422 на мусор).
- `GET /meta/positions?building_type_id=&q=` — позиции, реально присутствующие в живом
  перечне типа (комбобокс «+ стандарт»).

Три проверенных факта, на которых стоит контур:
1. **`building_type` достижим ТОЛЬКО через `segment.building_type_id`**, не через `position`
   (позиции висят на общем дереве `category`). Class scope: bt из сегмента; position/standard:
   bt из тела; position-scope исключает только классы ЭТОГО типа
   (`segment_id IN (SELECT id FROM segment WHERE building_type_id=:bt)`).
2. **Порядок блокировок listing→release ЕДИН во всех трёх мутациях** (сперва пишем listing,
   `ensure_open_release` — ПОСЛЕ) — без дедлока при конкурентной правке одного типа. Маркер
   создаётся **только при rowcount>0** (реальном изменении): no-op (нечего исключать /
   уже-живой класс) не плодит фантомный черновик на дашборде. Прямой тест
   `test_exclude_noop_returns_zeros_no_marker` это фиксирует.
3. **`listing_cell_chk` — это `RAISE EXCEPTION` (SQLSTATE `P0001`), НЕ нарушение ограничения.**
   asyncpg отдаёт его как `RaiseError`, SQLAlchemy оборачивает в `DBAPIError` — это **НЕ
   `IntegrityError`**. Хелпер `_is_cell_chk` ловит по `exc.orig.sqlstate == "P0001"` и
   транслирует в 409. 409 возможен только на add/restore (soft-delete триггер не ограничивает).

Ревью Opus подтвердило корректность scoped-WHERE и P0001-трансляции, но нашло, что
cross-building-type инвариант (самый критичный) не имел регрессионного стража в тестах —
код верен по инспекции, но `test_exclude_position_scale`/`test_meta_positions_scoped_*` прошли
бы и без фильтра. Fix-pass (`a439f05`) добавил второй building_type, делящий позицию, — тесты
теперь проваливаются, если фильтр убрать (re-review подтвердил эффективность). Плюс явный
restore→409 тест.

### Task 3 — типографика v3 + тексты на язык релиза (коммиты `e5b8552`, `a98276e`)

Чистые хелперы в `model.ts`: `splitQualifier` (голова/уточнение в скобках),
`excludeScaleForPosition`/`excludeScaleForStandard` (клиентский предрасчёт масштаба),
`pluralClasses`. Переписаны устаревшие тексты `excludedTooltip` (теперь принимает
`releaseLabel`) и `whereAllowedLegend` (остаётся **paramless**) на язык «войдёт в
следующий релиз» (старые «исключён в текущем черновике»/«показано текущее состояние»
противоречили исправленной модели). Типографика v3 в блоке — презентационно, без
изменения логики данных: имя стандарта medium+tracking-tight (приглушено свёрнутым,
foreground открытым), «все классы» — тихая строка с mint-иконкой `CheckCheck` (не чип),
уточнение позиции в скобках приглушено через `splitQualifier`.

### Task 4 — каркас режима правки (коммит `08fdabe`)

Локальный `editMode` (view — дефолт, полностью read-only) гейтит ВСЕ affordance:
тумблер соглашения `disabled`, alias `×`/«+ вариант» скрыты, `InlineEditText` получил проп
`readOnly` (пустое значение в view не рендерит ничего), «Объединить» только в edit.
Аккордеон «Где разрешён» — контролируемый в edit (`value=expanded`), uncontrolled в view;
вход в edit раскрывает все стандарты. **View-default сломал существующие тесты** мутаций/
инлайн-правки (кликали affordance, которых в view нет) — обновлены через `enterEditMode()`
перед взаимодействием, без ослабления ассертов.

### Task 5 — операции класс/позиция/стандарт + 5 ui-примитивов + kind (коммит `8ff56d4`)

5 стоковых shadcn-примитивов (`dialog`/`popover`/`command`/`checkbox`/`radio-group`) в
**унифицированной конвенции** `import { X as XPrimitive } from "radix-ui"` (не раздельный
`@radix-ui/react-*`, который генерит CLI) — сверено по `ui/dropdown-menu.tsx`. `command`
транзитивно тянет `input-group`/`textarea` (легитимные зависимости, не мёртвый код).
Операции (все под `editMode`): kind-дропдаун (читает `KIND_LABELS` — бейдж и опция не
дрейфуют), класс-`×` (исключение мгновенно, без диалога), «+ класс» (Popover+Checkbox),
позиция-`⊖` и kebab стандарта (`ExcludeDialog`), «вернуть». В edit роллап «все классы»
не применяется — всегда полный список чипов (`!editMode && isAllClasses(...) ? rollup : chips`).

**Двухфазный масштаб** (решение заказчика): диалог показывает клиентский предрасчёт из уже
загруженного дерева (мгновенно, без сети), ответ мутации возвращает **фактический** масштаб
в тост (истина при гонке); отдельного preview-эндпоинта нет. **Инвалидация 4 ключей**
(`vendor-where-allowed`/`vendor`/`matrix`/`dashboard`) у мутаций разрешений; `useAddAlias`/
`useRemoveAlias` НЕ трогаем (alias не в payload матрицы — решение заказчика).

### Task 6 — диалог «+ стандарт» (коммиты `3bda5fa`, `9e7816d`)

`AddStandardDialog` — три шага в одном `Dialog`: стандарт (`RadioGroup`, присутствующие
`disabled`+«уже присутствует») → позиция (`Command` combobox над `useMetaPositions`) →
классы (`Checkbox` над `useSegments`, одноклассовый тип → пометка). Пунктирная кнопка
«+ стандарт» внизу блока (edit); при присутствии во всех стандартах — `disabled`+тултип.

Ревью нашло **Critical**: `onAdd` делал fire-and-forget `.mutate()` и закрывал диалог
синхронно — при отказе POST диалог исчезал без единого сигнала (пользователь думал, что
добавилось), а проп `pending` был мёртв. Fix-pass (`9e7816d`): `await mutateAsync`, закрытие
только на успех, тост+остаётся открытым на ошибке (по образцу `confirmExclude`); + убран
`key`-remount (ломал exit-анимацию), сброс шагов перенесён в `onOpenChange`; + лоадер шага
классов; + тесты сабмита (успех: тело+закрытие; 409: диалог остаётся). Re-review подтвердил.

## Пять решений заказчика (были «открыто» в спеке — закрыты до реализации)

1. `other` → «прочее»: `KIND_LABELS` не меняем, дропдаун читает как есть (ноль дрейфа).
2. Исключение — ОДИН scoped-эндпоинт, не три ручки.
3. Масштаб N/M — двухфазно (preview=клиент, факт=сервер), без preview-эндпоинта.
4. Инвалидация алиасов — `["matrix"]` НЕ добавляем (alias не в payload матрицы).
5. `label` авто-open-релиза — нейтральный «рабочая версия», не ложь (`freeze_release` не
   трогает label). `freeze_release` в этом срезе не трогаем.

## Ловушки и уроки

### План соврал про токен: `text-success` не существует

Global Constraints плана мандатировал `text-success` / `--ds-state-success` для mint-иконки
«все классы». Такого токена в DS нет (есть только `--color-mint: #82D6CC` → `text-mint`), а
constraint «новых токенов не заводить» запрещал его создать. Контроллер поправил на реальный
`text-mint` (коммит `a98276e`) — интент (mint) и запрет на новые токены оба указывают на него.
Урок прежних срезов подтверждён: **планы врут в мелочах, ревью/реализация ловят.**

### Задачи гоняли pytest, но не ruff → E501 всплыл на финальном `just ci`

Бэкенд-задачи проверялись `pytest`+`mypy`+`just types` (как в брифах), но `ruff check` не
входил — длинные строки тестового кода (из verbatim-кода плана) прошли до финального гейта.
Ровно проектный урок «**перед пушем полный `just ci`, не ручной набор**»: имплементеру мало
прогнать релевантные тесты, нужен весь линт. На будущее — включать `ruff` в шаги бэкенд-задач.

### Ревью ловит тихие отказы, которые тесты happy-path не видят

Task 6 Critical (диалог «+ стандарт» глотал ошибку мутации) MSW-тесты не поймали — они всегда
успешны. Нашло целевое ревью через инспекцию (сравнение с `confirmExclude`). Тот же класс
(floating-promise без фидбэка ошибки) на exclude-операциях остался Minor в TECH_DEBT.

### Инвертированный TDD тут неприменим

Для `ensure_open_release` — обычный TDD (логика новая, RED до наката 0006), в отличие от db-
тестов существующих вьюх/функций (там ждём PASS с первого прогона). Различие важно: FAIL
db-теста существующей логики = ошибка в тесте; FAIL нового — ожидаем до реализации.
