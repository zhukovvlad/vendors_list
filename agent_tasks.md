# ТЗ для агента: приложение учёта вендор-листов

Заказчик: девелопер (заказчик строительства). Система учитывает перечни
одобренных производителей (вендор-листы) по типам объектов и классам зданий,
а также фиксирует выбор вендоров на проектах и подсвечивает отступления.

---

## 0. Что УЖЕ сделано (не переделывать)

База данных спроектирована, реализована и проверена на PostgreSQL 16.
Два файла, накатываются строго в этом порядке:

1. **`vendor_list_schema.sql`** — ядро стандартов (схема `public`):
   справочники (типы объектов, классы/группы, дерево разделов, позиции,
   вендоры + синонимы, соглашения), живой перечень `listing` с аудитом
   `change_log`, издания `release` + неизменяемые снимки `release_listing`,
   функция `freeze_release`. Встроены: контроль целостности ячеек
   (уникальность вендора и мета-строки, запрет смешивания «вендоры +
   требование»), звёздочка `*` = активное соглашение (`vendor_starred()`,
   заморозка в снимок, живая вьюха `listing_live`), аудит соглашений.
2. **`compliance_module.sql`** — модуль соответствия (схема `compliance`):
   проекты, выбор вендоров, светофор, сводка.

Ядро — источник истины. Схему НЕ генерировать заново из ORM-моделей.

## 1. Бизнес-правила модуля соответствия (уже зашиты в SQL — знать при разработке UI/API)

Светофор по позиции (`position_state` во вьюхе `compliance.project_position_status`):

- 🟢 `compliant` — в стандарте есть список разрешённых для класса проекта,
  выбор сделан, ВСЕ выбранные вендоры из списка. **Строгое правило:
  один лишний вендор → вся позиция красная.**
- 🔴 `deviation` — список есть, но хотя бы один выбранный вендор вне списка.
- 🟡 `manual_check` — списка нет: в ячейке требование («Россия», ГОСТ,
  «по согласованию») или прочерк «–». Машина не судит; текст требования —
  в колонке `standard_requirement`. Нужна ручная проверка человеком.
- ⚪ `open` — позиция в области стандарта, выбор ещё не сделан.

Процент соответствия (`compliance_pct` в `compliance.project_summary`) =
🟢 / (🟢 + 🔴). Жёлтые и открытые в процент не входят, показываются
отдельными счётчиками.

Прочее:
- Проект привязан к ОДНОМУ классу (`segment_id`). Поле `release_id` —
  справочное, на расчёт не влияет.
- На позицию можно выбрать НЕСКОЛЬКО вендоров; ЛЮБОГО из справочника
  (в т.ч. вне стандарта) — отступления заносятся и подсвечиваются,
  а не запрещаются.
- Соответствие вычисляется против ЖИВОГО `listing`: изменили стандарт —
  флаги пересчитались сами.
- Бренд-ключ: представитель бренда (ИСТРАТЕХ → Grundfos) засчитывается
  как сам бренд (`compliance.brand_key`).
- У выбора есть `rationale` (основание отступления/согласование) и
  `source_ref` (лист DWG / раздел PDF, откуда взято).

## 2. Стек (согласован)

- **БД**: PostgreSQL 16.
- **Бэкенд**: Python 3.12+, **FastAPI** (REST + автогенерируемый OpenAPI),
  Pydantic v2 для схем запросов/ответов, `uv` или Poetry для зависимостей.
- **Слой доступа к БД**: schema-first — базой владеют SQL-миграции, а не
  модели. **SQLAlchemy Core** (Core, НЕ declarative-ORM) либо asyncpg +
  сырой SQL. Полноценный ORM (SQLAlchemy ORM, Django ORM, Prisma и т.п.)
  как владелец схемы — ЗАПРЕЩЁН: триггеры/функции/CHECK-и он не выражает.
- **Миграции**: чистый SQL через Alembic (`op.execute` / `*.sql`-скрипты)
  или dbmate. Два существующих файла — базовые миграции №1 и №2,
  autogenerate по моделям НЕ использовать.
- **Типы для фронта**: из OpenAPI-схемы FastAPI генерировать TS-типы
  (`openapi-typescript`) в CI — сквозная типизация без общего рантайма.
- **Фронтенд**: Vite + React + TypeScript, shadcn/ui + Tailwind,
  TanStack Table (матрицы перечня), TanStack Query, react-hook-form + Zod.

## 3. Задачи бэкенда

1. **Транзакционная обёртка с идентичностью** (критично для аудита):
   FastAPI-dependency, открывающая транзакцию и первым делом выполняющая
   `SET LOCAL app.user = '<логин из SSO>'` (значение — параметром,
   не конкатенацией). Именно `SET LOCAL` (не `SET`) — при transaction
   pooling (PgBouncer) обычный SET протекает между запросами и аудит
   подпишется чужим именем. Все пишущие эндпоинты работают только
   через эту dependency.
2. **Аутентификация/авторизация**: корпоративный SSO/OIDC (Entra ID /
   Keycloak) через `authlib`; проверка ролей — FastAPI-dependencies.
   Свою аутентификацию не писать. Роли в API, не в БД:
   - `viewer` — любой аутентифицированный: чтение стандартов, изданий,
     проектов и отчётов соответствия;
   - `admin` — редактирование стандартов, фиксация изданий
     (`freeze_release`), управление соглашениями и проектами.
3. **API поверх готовых объектов БД**: живой перечень — из `listing_live`;
   выгрузка издания — из `release_listing`; светофор и сводка — из
   `compliance.project_position_status` / `project_summary`. Бизнес-логику
   расчётов в коде не дублировать — она уже в БД.
4. **Импорт Excel** (самая трудоёмкая задача, НЕ пытаться сделать
   полностью автоматическим; openpyxl/pandas — сильная сторона выбранного
   стека):
   - этап 1: загрузка файла в staging-таблицы как есть;
   - этап 2: экран сверки — распознанные вендоры мапятся на канонический
     справочник через `vendor_alias` + pg_trgm (нечёткое совпадение),
     человек подтверждает маппинг;
   - этап 3: коммит в `listing` одной транзакцией.
   Ловушки исходных файлов: неконсистентные написания вендоров, хвостовые
   пробелы, `*` у имён (= соглашение, вести через `agreement`, в текст
   имени не тащить), `(Native)`, объединённые ячейки шапок, мусорные
   колонки (один лист сообщает 16384 колонки), лист с датой 2022 года.
5. **Экспорт издания в Excel** — воспроизведение перечня из
   `release_listing` (денормализованные подписи там ровно для этого).

## 4. Задачи фронтенда (приоритет — просмотр: большинство пользователей только читает)

1. Матрица перечня: строки — позиции по дереву разделов, колонки — классы,
   в ячейках вендоры (звёздочка из `vendor_starred`), фильтры/поиск.
   TanStack Table, серверная пагинация/фильтрация.
2. Карточка проекта: светофор по позициям (цвета `position_state`),
   счётчики 🟢/🔴/🟡/⚪, `compliance_pct`, список лишних вендоров,
   у жёлтых — текст требования (`standard_requirement`).
3. Занесение выбора: позиция → вендоры (мультивыбор из справочника,
   любой вендор), поля `rationale`, `source_ref`.
4. Админ-контур: редактирование перечня (с внятным показом ошибок
   триггеров целостности), фиксация издания, управление соглашениями.
5. Экран импорта из п. 3.4 (staging → сверка → коммит).

## 5. Порядок работ

1. Каркас: репозиторий backend (FastAPI, uv/Poetry) + frontend (Vite),
   CI, Alembic/dbmate с двумя базовыми SQL-миграциями, генерация
   TS-типов из OpenAPI.
2. Транзакционная обёртка + SSO + RBAC (задачи 3.1–3.2).
3. Read-only API + матрица перечня (просмотр — главный сценарий).
4. Модуль проектов: занесение выбора + светофор.
5. Импорт Excel (staging → сверка → коммит), затем экспорт изданий.
6. Админ-редактирование перечня и изданий.

## 6. Чего НЕ делать

- Тест для любой новой логики: «должно ли это выполняться, даже если в базу
  напишут в обход нашего API?» Да (инвариант, определение статуса) — место
  в БД, миграцией. Нет (экран, процесс, интеграция) — место в бэкенде.
- Не генерировать схему из ORM; не «чинить» триггеры/CHECK-и переносом
  логики в приложение.
- Не хранить вычислимое: звезда вендора и статусы соответствия — только
  из функций/вьюх БД.
- Не заводить таблицы пользователей/ролей в БД (пока ролей две и личности
  в SSO). Пересмотреть, если появятся роли, зависящие от данных, или
  внешние пользователи.
- Не делать импорт Excel полностью автоматическим — маппинг вендоров
  подтверждает человек.
- Не смешивать файлы: ядро и модуль соответствия остаются двумя
  независимыми миграциями.
