# 2026-07-15 — Рестайл матрицы «Каталог стандартов» в языке ReviewItemsTable

Ветка `feat/matrix-restyle` от чистого main. Заказчику понравилась таблица
`ReviewItemsTable` из соседнего проекта UDP (плотная grid-таблица с sunken-шапкой,
чип-языком ячеек, sticky-колонкой). Разведка на брейншторме (см. спека) дала три
вывода: наша матрица — **pivot** (позиции × классы), а не плоский список атрибутов;
у позиции/стандарта нет собственных атрибутов, оправдывающих отдельный экран-каталог;
пункт сайдбара «Каталог стандартов» уже ведёт на `/matrix`. Значит правильный ход —
перенести визуальный **язык** `ReviewItemsTable` на существующую матрицу, не плодя
второй экран. Планирование — 6 задач (TDD где применимо), исполнение по шагам плана.

## Что сделано

Визуальный рестайл экрана матрицы «Каталог стандартов» (`/matrix`), без единой
строчки backend/миграций/новых маршрутов/инлайн-правки — срез строго read-only:

- **Requirement-бейдж → warning-tan.** `Badge variant="requirement"` был
  `bg-muted text-muted-foreground italic` (приглушённый курсив); стал
  `border-warning/30 bg-warning/10 text-warning` — читается как «внимание,
  здесь особое требование», не как выключенный текст.
- **`splitQualifier` поднят в `frontend/src/lib/qualifier.ts`.** Раньше жил
  только в `screens/vendors/model.ts`; матрица стала вторым потребителем, так что
  функцию вынесли в общий модуль с тестом, а `vendors/model.ts` оставил
  ре-экспорт (`export { splitQualifier } from "@/lib/qualifier"`) — импорты
  вендор-экранов (`PositionRow.tsx`/`WhereAllowedSection.tsx`) не тронуты.
- **Чип-язык ячеек в `renderCell`** (`columns.tsx`): вендор — компактный чип с
  бордером; действующее соглашение (`starred`) — mint-заливка + ★ (`chart-2`,
  `aria-label="действующее соглашение"` сохранён); `ujin_integration` —
  приглушённая приписка «Ujin» за левым бордером-разделителем; требование
  (`spec_text`) — новый warning-бейдж; пусто — тихий `—`
  (`text-muted-foreground/60`). Приоритет веток (вендоры → `spec_text` → `—`)
  не менялся — это инвариант БД (см. «Ключевые решения» ниже), не выбор дизайна.
- **Двухстрочная колонка «Позиция»** — голова/уточнение в скобках разнесены на
  две строки через `splitQualifier` (уточнение — приглушённым мелким текстом
  под головой).
- **Sticky-left колонка «Позиция» + вертикальные разделители групп** — колонка
  «Позиция» липнет `position: sticky; left: 0` с непрозрачным фоном и
  `group-hover`-подсветкой (своя, потому что opaque-фон маскирует
  `hover:bg-muted/50` строки); первый лист каждой группы классов несёт левый
  бордер-разделитель (первая группа — без бордера, чтобы не удваивать линию
  вплотную к `border-r` липкой «Позиции»).
- **Sunken uppercase-шапка таблицы** (`bg-muted [&_th]:uppercase`), лёгкая
  шапка экрана (заголовок «Каталог стандартов» + подзаголовок) над
  фильтр-`Card`.
- **Строки-разделы (`category_path`) в две ячейки**: sticky-left ячейка с
  подписью раздела (uppercase, tracking) + пустой `colSpan={leafCount - 1}` —
  иначе при горизонтальном скролле подпись раздела «уезжала» вместе с телом
  таблицы, а не оставалась на месте вместе с «Позицией».
- **Плотность**: сниженные вертикальные паддинги ячеек (`[&_td]:py-1.5`) и
  `group`-класс на строке для согласованного hover эффекта строки + липкой
  ячейки.

## Ключевые решения

- **Стиль поверх существующего pivot, не переписывание TanStack.** Фиксированный
  CSS-grid из `ReviewItemsTable` тут неприменим — число колонок-классов
  динамическое (server pivot). Переносился только визуальный язык поверх уже
  работающей `useReactTable`-таблицы; сама pivot-механика (данные, колонки,
  URL-контракт фильтров, пагинация) не тронута.
- **Sticky-left без sticky-top — сознательно вне среза.** Ревью спеки против
  кода нашло: `overflow-x-auto` на обёртке shadcn `Table` делает div
  горизонтальным скролл-контейнером; `sticky top-0` липнет именно к нему, а без
  явного `max-height` контейнер вертикально не скроллит вовсе — sticky-top
  оказывается инертным (тот же баг был и в HTML-мокапе, спека главнее мокапа).
  Канонический фикс — `max-height` + внутренний вертикальный скролл, но это уже
  смена layout (фиксированная высота таблицы рядом с пагинацией), а не
  косметика поверх существующего экрана. Решение: в этот срез берём только
  sticky-left (снимает главную боль широкого pivot — видимость колонки
  «Позиция» при горизонтальном скролле классов). Sticky-top — кандидат в
  отдельный срез с осознанной сменой layout.
- **Чипы ячеек — экранные утилитарные классы в `columns.tsx`, не DS-компоненты.**
  Золотое правило: `components/ui/` — только shadcn-примитивы. Чип-язык матрицы
  специфичен для этого экрана (вендор-чип/mint-звезда/Ujin-приписка) — не
  переиспользуемый DS-примитив, значит остаётся локальным.
- **`splitQualifier` → `lib/qualifier.ts` с ре-экспортом.** Как только у чистого
  презентационного хелпера появился второй потребитель (позиция матрицы, помимо
  блока «Где разрешён» на карточке вендора), решение — поднять в общий модуль,
  не дублировать функцию и тест. Ре-экспорт в `vendors/model.ts` держит blast
  radius минимальным — существующие импорты вендор-экранов не переписывались.
- **Requirement-бейдж → warning-tan.** Вариант использовался только в матрице
  плюс один smoke-тест — перекрас безопасен, подтверждено ревью спеки перед
  реализацией.
- **`ColumnMeta` module augmentation.** TanStack Table по умолчанию не
  типизирует `meta` — без `declare module "@tanstack/react-table" { interface
  ColumnMeta<TData, TValue> { className?: string; headerClassName?: string } }`
  в `matrix/model.ts` `tsc` падал бы на обращении `column.columnDef.meta?.className`
  в `columns.tsx`/`MatrixScreen.tsx`. Augmentation — единственный способ
  протащить per-column CSS-классы (sticky/бордеры групп) через типобезопасный
  API без форка библиотеки.
- **Приоритет vendors → spec_text → «—» в `renderCell` — инвариант БД, не выбор
  дизайна.** Триггер `listing_cell_chk` (`0001_core_schema.sql`) запрещает
  ячейке одновременно нести вендоров и `spec_text` — ячейка listing это ЛИБО
  вендоры, ЛИБО мета-текст требования, никогда оба сразу. Значит ветка «показать
  оба» не существует как случай и реализовывать её не нужно; порядок веток в
  `renderCell` сохранён как есть.

## Отложено / вне среза

- Sticky-top шапки таблицы (требует `max-height` + внутренний скролл — смена
  layout, не косметика).
- Инлайн-правка ячеек матрицы (ячейка = множество вендоров, add/remove на
  пересечении — фича уровня ТЗ §6, не этот срез).
- `tabular-nums` из исходной спеки — не применимо: в ячейках матрицы имена
  вендоров и текст требований, числовых колонок нет; записано в плане явно, а
  не пропущено молча.
