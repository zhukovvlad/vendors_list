## Протокол на каждый таск
1. Выдай субагенту РОВНО текст таска из плана (блоки Files / Interfaces / Steps).
   Напомни: строгий TDD по шагам — failing-тест → прогон (падает как описано) →
   минимальная реализация → прогон (проходит) → коммит. Ничего сверх таска.
2. Получив результат — проверь: тесты реально гоняли и они зелёные; дифф
   соответствует плану; типы/имена совпадают с блоком Interfaces соседних тасков.
3. Прогони сам: `cd backend; uv run pytest <файлы таска> -q` и, где есть типы,
   `uv run mypy app`. Гейт: НЕ переходить к таску N+1, пока N зелёный и отревьюен.

## Критичные инварианты (не дать субагенту нарушить)
- Только **SQLAlchemy Core** (async, asyncpg). НЕ ORM. Схема — источник истины;
  ничего вычислимого не пересчитываем (светофор/звезда/проценты — вьюхи/функции БД).
- Пишущая транзакция ПЕРВЫМ запросом: `set_config('app.user', :login, true)`,
  логин **bind-параметром**, не конкатенацией. `created_by`/аудит/инвариант ячейки —
  на триггерах БД, вручную НЕ проставлять.
- Сброс при перезапуске — **DELETE в FK-порядке, НЕ `TRUNCATE CASCADE`** (иначе
  затрёт `compliance.project*`). `--force` снимает только guard; FK RESTRICT
  физически защищает проекты.
- `building_type`/`segment`/`segment_group` уже засеяны миграцией `0001` — не
  создавать, только lookup по имени.
- Маркеры требований (whole-cell) — ровно `Россия` и `По согласованию…` (префикс);
  `ГОСТ` НЕ маркер. Смешанная ячейка `Россия, <вендоры>` → список вендоров +
  `Требование: Россия` в `listing.note` каждого (не делать `Россия` вендором).
- Дедуп вендора В ПРЕДЕЛАХ ЯЧЕЙКИ обязателен (uq_listing_cell_vendor) — реализован
  в build_load (Task 7).
