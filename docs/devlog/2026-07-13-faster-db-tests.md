# 2026-07-13 — Ускорение локальных db-тестов (session-scoped соединение)

Ветка `chore/faster-db-tests` от `main` (после мержа PR #23). Мотив: локальный
`uv run pytest -q` шёл ~9:24 (564с) на 178 тестах — больно на итерациях.

## Диагноз: дело в RTT до Neon, не в тестах

`conftest.py` открывал **новое соединение на каждый тест** (`engine` function-scoped +
`NullPool`). Стоимость = RTT до БД × (TLS-хендшейк + запросы). Ключевое наблюдение —
сравнение с CI:

| Где | Прогон 178 тестов | Причина |
|-----|-------------------|---------|
| Локально (регион +0300 → Neon US/EU) | **564с** (~3.2с/тест) | RTT ~100-300мс: один TLS-хендшейк на тест = ~0.5-1.5с, платится 178 раз |
| GitHub CI (раннер рядом с Neon) | **109с** (~0.6с/тест) | RTT ~1-5мс: хендшейк ≈ 10мс, сетевое время near-zero |

Тот же код, та же удалённая БД — разница ~5x чисто из-за географической дистанции до
Neon. То есть медленно было **только локально**; CI и так быстрый.

## Как сделано у соседей (и что применимо)

- **CIW**: БД в тестах нет вообще — гексагональная архитектура, in-memory `Fake*` через
  DI. Нам нельзя: золотое правило — БД источник истины, тестируем реальные вьюхи/функции/
  триггеры (`vendor_where_allowed`, `ensure_open_release`, `listing_cell_chk`).
- **UDP**: реальная БД, но session-scoped engine + savepoint-изоляция
  ([conftest.py:135](../../../UDP/backend/tests/conftest.py)), `join_transaction_mode=
  "create_savepoint"`. UDP синхронный — у него нет нашей проблемы с event-loop. Паттерн
  переносим на async.

## Реализация (async-версия UDP-паттерна)

- **`pyproject.toml`**: `asyncio_default_fixture_loop_scope = "session"` +
  `asyncio_default_test_loop_scope = "session"` (pytest-asyncio 1.4 поддерживает оба).
  Единый event loop на сессию — иначе соединение, созданное на session-loop, недоступно
  из function-loop теста («attached to a different loop»). Это ровно то ограничение,
  из-за которого раньше стоял `NullPool` + function-scope.
- **`conftest.py`**: `engine` и новое `_shared_conn` — `scope="session", loop_scope=
  "session"` (одно физическое соединение к Neon на всю сессию). `db_conn` (function) —
  `await _shared_conn.begin()` → yield → `rollback()`: своя верхнеуровневая транзакция на
  тест поверх общего соединения. `NullPool` оставлен — пул не нужен, держим единственное
  соединение сами. `tx`-override (SAVEPOINT на запрос) композится: savepoint под
  транзакцией теста.

Изоляция та же (rollback на тест), но хендшейк платится один раз, не 178.

## Результат

`178 passed in 175.87s (2:55)` — **было 564с (9:24), стало 176с: ~3.2x локально.**
Дальше упираемся в per-query RTT (его убрал бы только локальный Postgres — но тогда
теряем прогон против самой Neon-ветки; сознательно не делаем). CI не трогали по сути —
там эффект мал (RTT и так низкий), но и не сломали: паттерн (одно клиент-соединение,
транзакция на тест) дружелюбен к транзакционному пулеру Neon (`statement_cache_size=0`
сохранён; между тестами нет открытой транзакции → нет idle-in-transaction).

## Ловушки

- **Session-loop обязателен для ОБОИХ** (fixtures И tests). Только fixture-scope не хватит:
  тест на function-loop не сможет `await` на соединении, созданном на session-loop.
- **Транзакционный пулер Neon (PgBouncer):** паттерн совместим — соединение держим одно,
  сервер-бэкенд назначается на транзакцию и освобождается на rollback; между тестами
  транзакции нет. `statement_cache_size=0` обязателен (prepared statements не переживают
  пулер) — оставлен как был.
- **Не общий connection-savepoint (как в SQLAlchemy-рецепте), а транзакция-на-тест.**
  Проще и ближе к прежней структуре: убрали только переоткрытие соединения, семантику
  изоляции не меняли.
