## Зачем

Разовый сид ([devlog](2026-07-09-excel-seed-import.md)) писал ~14k строк в одной транзакции
**построчными** `INSERT … RETURNING id` — каждый оператор отдельным round-trip до Neon
(us-east-1), поэтому реальный прогон длился **минуты** и держал широкое окно для фризов
(инцидент с `idle in transaction`). Цель — минуты → секунды, при сохранении атомарности §14
(один прогон = полная перезагрузка, всё-или-ничего) и всех инвариантов db-тестов.

## Что сделано (4 задачи, TDD)

- **`_apply_timeouts(conn)`** — два `SET LOCAL` (`statement_timeout='60s'`,
  `idle_in_transaction_session_timeout='15s'`), вызывается в `run()` внутри `begin()` до
  `execute`. `idle=15s` безопасен ТОЛЬКО потому, что `build_load` (весь парсинг Excel) идёт
  **до** транзакции — внутри неё клиентских пауз нет.
- **`_prealloc_ids(conn, table, n)`** — предвыделение `n` id из sequence таблицы через
  `nextval(pg_get_serial_sequence(:t,'id'))` над `generate_series(1,:n)`. НЕ `RETURNING`
  (порядок строк `INSERT … RETURNING` формально не документирован), НЕ `setval` (id взяты из
  того же sequence — он уже «впереди»). `nextval` нетранзакционен, дырки в id при откате
  легальны.
- **Батч category/position/vendor/agreement** — по одному `executemany` с явными id из карт
  `cat_id`/`pos_id`/`vendor_id`. Категории идут строго **родители-раньше-детей**
  (`plan.categories == tree.ordered()`, лексикографический порядок кортежа-пути): asyncpg
  `executemany` исполняет наборы последовательно в порядке списка — это (а не `DEFERRABLE`)
  держит FK `parent_id` валидным построчно.
- **`listing` → один `executemany`** (~13k строк) — валидация segment-ключа вынесена в
  предпроход (посреди batch валидировать нельзя): собираем `listing_params`, на промахе ключа
  `raise SeedError` со списком доступных классов, затем один batch. Триггеры
  `listing_stamp`/`listing_audit`/`listing_cell_chk` срабатывают построчно и под `executemany`
  — аудит и инвариант ячейки сохранены.
- **TECH_DEBT** — отмечен сознательно непокрытый автотестом эффект таймаутов.

## Верификация — замер реального прогона

**Полный сид 3 стартовых Excel по новой батч-схеме против тест-ветки Neon: `real 31.4s`**
(end-to-end, включая парсинг Excel). Было (построчный путь) — минуты. **Цель «минуты →
секунды» подтверждена числом.**

Масштаб прогона (всё в одной транзакции, закоммичено успешно): ~13 244 `listing`
(residential 7590 / office 4545 / social 1109), из них 2255 со звёздным вендором; 68 категорий,
743 позиции, 898 уникальных вендоров, 117 соглашений. Поведение идентично старому построчному
пути на реальных данных: сработали все ожидаемые предупреждения (in-cell дедуп
`uq_listing_cell_vendor` — Flamco/System Air/Ронсон/Newton/СВОК/Аданат; смешанные ячейки
`Требование: Россия`; конфликты имён категорий между файлами). `_apply_timeouts` на обрыв не
сработал — прогон уверенно уложился в 60s statement / 15s idle.

> Прогон — против **тест-ветки** Neon (`DATABASE_URL=$DATABASE_URL_TEST`), рабочую БД не
> трогали. Замер сделан до мержа, чтобы подтвердить цель числом. Закоммиченные данные затем
> убраны с тест-ветки (`loader._reset`), чтобы не ломать регрессионный db-набор.

- **db-тесты** ([tests/db/test_seed_loader.py](../../backend/tests/db/test_seed_loader.py)) —
  9/9 PASS на тест-ветке Neon (6 исходных + 3 новых гварда: вложенные категории → `parent_id`,
  sequence-ahead после явных id, применение GUC через `SHOW`). Инвертированный TDD: логика БД
  уже есть, ждём PASS.
- `just ci` зелёный локально; CI на PR #7 зелёный (backend db-тесты на **эфемерной** Neon).

## Решения и нюансы

- **Атомарность §14 сохранена** — один `begin()` в `run()`, промежуточных commit нет, любой
  `raise` (в т.ч. из предпрохода валидации listing) откатывает всю транзакцию. Сброс — `DELETE`
  в FK-порядке, `compliance.*` не трогается; FK `RESTRICT` из проектов физически не даёт снести
  ядро «из-под» проекта даже с `--force`.
- **Предпроход валидации listing** повторяет прежний контракт ошибки байт-в-байт (тот же
  `SeedError`, тот же отсортированный список доступных классов), просто исполняется до первого
  INSERT.
- **`strict=True` на `zip`** предвыделённых id с объектами (сверх плана) — тихое рассогласование
  длин превращается в громкий `ValueError`; безопасно, т.к. `_prealloc_ids` возвращает ровно `n`.
- Финальное ревью всей ветки — 0 Critical/Important; 2 Minor (латентная привязка `_prealloc_ids`
  к serial-sequence; косметическая аннотация) отложены как несущественные.

## Хвост: хрупкость теста, вскрытая замером → PR [#8](https://github.com/zhukovvlad/vendors_list/pull/8)

Боевой замер вскрыл предсуществующую слабость `test_listings_page_and_filter`: он искал свою
фабричную строку среди **первых 100** результатов сегмента (`limit=100`). Приложение штатно
живёт с заполненной БД — на тысячах реальных `listing` фабричный `List-V` уходит за первую
страницу, и тест краснеет. Это реальная мина под CI (сработала бы, как только родительский
Neon-проект засеют стандартами). Фикс (ветка `fix/listings-test-baseline-independent`, отдельно
от write-path): свою строку изолируем фильтром по **свежесозданной позиции** (`position_id`,
который сид не трогает) — проверка независима от заполнения БД; сегментный запрос сохранён для
пагинации/`total`. Blast radius — только этот тест (freeze/projects/db-тесты scoped к своим
фикстурам либо `execute()` сбрасывает таблицы первым делом).

## Что осталось / открытые хвосты

- **`_prealloc_ids` привязан к `pg_get_serial_sequence`/serial-колонкам** — сломается, если id
  когда-либо переведут на `GENERATED ALWAYS AS IDENTITY` (потребует `OVERRIDING SYSTEM VALUE`).
  Сейчас корректно, покрыто db-тестами; заметка на случай миграции схемы.
- **Эффект таймаутов не покрыт автотестом** — тест проверяет их ПРИМЕНЕНИЕ через `SHOW`, но
  реальный обрыв без медленного оператора (`pg_sleep`) не воспроизводится (в TECH_DEBT).
- Интерактивный импорт (staging → сверка → коммит) и экспорт изданий — этап §5, отдельная
  задача; батч-путь этого разового сида к ней не относится.
